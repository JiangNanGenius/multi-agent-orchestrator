# 多Agent智作中枢现代中文多 Agent 协作架构改造完整计划

## 一、目标定义

本次改造的目标，是在**保留现有核心工作流价值**的前提下，将项目升级并统一命名为 **多Agent智作中枢**，同时把当前“固定角色、固定命名、固定刷新方式”的实现，升级为一套**现代中文命名、可扩展 Agent 注册、可自动生成完整 SOUL、可自动部署、且刷新机制更稳定**的多 Agent 协作系统。

本次改造**不推翻原有取数思路**，重点处理的问题是：看板命名过于封建、执行角色不易扩展、SOUL 维护成本高、任务刷新与回显机制不稳定，以及 OpenClaw 串行约束下的调度体验不佳。

## 二、必须遵守的设计约束

### 1. 命名与组织方式

系统整体改为**现代中文命名**，不再使用朝廷、封建、君臣等表达。

新的组织结构如下：

| 层级 | 名称 | 说明 |
|---|---|---|
| 一级入口 | 总控中心 | 统一受理任务、首轮判断、快速处理、异常升级 |
| 流程节点 | 规划中心 | 负责需求拆解、方案整理、任务结构化 |
| 流程节点 | 评审中心 | 负责质量审查、补充约束、决定是否放行 |
| 流程节点 | 调度中心 | 负责任务分派、状态跟踪、结果汇总 |
| 执行域 | 专业执行组 | 专家 Agent 的容器层 |
| 执行单元 | N 个 XX 专家 | 如代码专家、测试专家、数据专家、运维专家、部署专家、文案专家等 |
| 系统运维 | 管理中心 | Agent 配置、模型设置、权限、注册管理 |
| 定时能力 | 搜索专家 | 全网搜索简报、订阅、定时任务类能力 |
| 任务对象 | 任务单 | 统一使用任务单作为正式对象命名 |
| 结果对象 | 结果报告 | 统一使用结果报告作为正式输出对象命名 |
| 主看板 | 协作中枢 | 原“军机处看板”统一改称协作中枢 |
| 协作界面 | 协同讨论 | 原“朝堂议政”统一改称协同讨论 |

### 2. SOUL 体系约束

每个角色仍然保留**一整份完整的 `SOUL.md`**，不拆解为碎片提示词拼接。

但是新增 Agent 时，不再直接手写整份 SOUL，而是遵循：

> **先生成 Agent Registry 规范，再由 Agent 根据规范自动生成完整 SOUL。**

也就是说，Registry 是输入规范，SOUL 是最终产物。SOUL 仍是独立完整文档，但来源改为规范驱动的自动生成流程。

### 3. 自动部署约束

新增 Agent 必须支持**自动部署**，且规范要尽量让 Agent 自己也能看懂和执行。

新增 Agent 的标准流程为：

1. 注册 Agent 规范；
2. 自动创建目录；
3. 自动生成完整 `SOUL.md`；
4. 自动写入接入配置与映射；
5. 自动让看板识别；
6. 自动完成最小可用校验。

### 4. 总控中心实时性约束

总控中心必须保持**最高实时性**。

默认情况下，总控中心只做以下高时效动作：

| 类型 | 是否由总控中心直接处理 |
|---|---|
| 任务识别与分流 | 是 |
| 轻量修复 | 是 |
| 配置补齐 | 是 |
| 简单问题定位 | 是 |
| 明确且短路径的问题处理 | 是 |
| 复杂规划 | 否，转规划中心 |
| 多 Agent 协作任务 | 否，转标准流程 |
| 长时间执行任务 | 否，避免占用实时链路 |

只有在进入**系统性修复模式**时，总控中心才允许执行较长时间的系统级变更任务。

### 5. 系统性修复边界

“系统性修复”仅指以下系统级变更：

| 属于系统性修复 | 不属于系统性修复 |
|---|---|
| 修改 SOUL | 制作 Skill |
| 修改看板对接 | 测试 Skill |
| 修改系统接入关系 | 普通业务执行 |
| 修改系统级协作逻辑 | 常规内容处理 |
| 修改注册与部署链路 | 一般专家任务 |

这条边界会直接写入总控中心的 SOUL、注册规范和调度策略中。

### 6. OpenClaw 串行处理约束

必须严格服从 OpenClaw 的现实限制：

> **单个 Agent 在上一个任务未完成之前，下一个任务无法开始。**

因此后续架构设计不做单 Agent 并发假设，而是采用：

- **多 Agent 横向扩容**；
- **单 Agent 串行执行**；
- **总控中心快速卸载任务**；
- **通过新增 N 个 XX 专家来分摊任务，而不是把更多任务堆给同一个 Agent**。

## 三、核心改造方向

### 1. 中文命名与界面信息架构重构

目标是彻底去掉封建叙事，但保留原有流程骨架的纪律性。

需要修改的内容包括：

| 范围 | 改造内容 |
|---|---|
| 前端页面标题 | 全量切换到现代中文命名 |
| 面板名称 | 与新组织结构保持一致 |
| 任务详情文案 | 任务单、结果报告、状态流等语义统一 |
| 角色文案 | 全量从旧角色映射到新角色 |
| 文档与说明 | README、示例、说明文档同步更新 |
| SOUL 中协作表述 | 改为现代中文组织术语 |

### 2. Agent Registry 规范设计

Registry 是新增 Agent 的唯一入口，也是 SOUL 自动生成和自动部署的前提。

Registry 文件将采用**强结构化中文 + 稳定字段**设计，便于人和 Agent 共同理解。

建议字段如下：

| 字段组 | 字段内容 |
|---|---|
| 基础信息 | agent_id、中文名称、分组、专家类型 |
| 能力定义 | 擅长任务、边界限制、优先级 |
| 协作关系 | 上游、下游、可协作对象、禁止越权对象 |
| 输入输出 | 输入格式、输出模板、状态更新要求 |
| 刷新要求 | 心跳频率、进度上报时机、停滞判定 |
| 部署要求 | 目录、配置注入、是否自动接入看板 |
| SOUL 参数 | 语气、职责强度、工作流要求、命令规范 |
| 调度信息 | 所属执行域、是否可被总控中心直派、是否允许系统级任务 |

### 3. 完整 SOUL 自动生成机制

每个 Agent 继续保持独立完整 SOUL，但来源改成规范驱动生成。

生成后的 SOUL 结构建议统一为以下章节：

| 章节 | 内容 |
|---|---|
| 角色定位 | 说明该 Agent 的职责和位置 |
| 核心职责 | 必须完成的工作 |
| 任务受理规则 | 能接什么任务、不能接什么任务 |
| 标准处理流程 | 从接收、分析、执行、回传的全过程 |
| 刷新与回报规范 | 何时必须刷新状态、回传进展 |
| 协作规则 | 可联系对象、转交规则、升级规则 |
| 异常处理 | 卡住、信息不足、失败时的动作 |
| 输出要求 | 回传格式、摘要要求、结构要求 |
| 风格与语气 | 保持可读性和执行一致性 |

总控中心的 SOUL 会额外加入：

- 高实时性约束；
- 可直接处理任务边界；
- 系统性修复模式进入条件；
- 不得长期占用处理链路的规则。

### 4. 自动部署链路建设

新增 Agent 后，系统应能尽量自动完成接入，减少手工修改。

自动部署链路目标如下：

| 步骤 | 动作 |
|---|---|
| 1 | 读取 Registry 规范 |
| 2 | 创建 `agents/{agent_id}/` 目录 |
| 3 | 生成并写入完整 `SOUL.md` |
| 4 | 更新 Agent 聚合配置与映射 |
| 5 | 更新看板识别配置 |
| 6 | 注入默认刷新与状态规则 |
| 7 | 运行最小校验并返回部署结果 |

### 5. 刷新机制升级

本次引擎改造重点不是重写取数，而是修正刷新链路。

已知问题方向包括：

| 问题 | 表现 |
|---|---|
| 全量刷新过重 | 页面频繁拉取整批数据，造成资源浪费与延迟 |
| 多面板重复请求 | 同一数据被多个面板重复获取 |
| 操作后回显慢 | 状态更新后 UI 反馈不及时 |
| 局部更新能力弱 | 卡片、详情、统计之间容易不同步 |
| 轮询与实时更新混杂 | 刷新来源不统一，互相覆盖 |
| 队列与占用不可见 | 串行约束下等待原因不清晰 |

升级后的目标是：

| 目标 | 方案 |
|---|---|
| 统一刷新调度 | 由统一刷新调度器协调所有刷新入口 |
| 面板共享数据源 | 避免重复请求 |
| 增量更新 | 优先局部刷新，而不是整页重拉 |
| 操作后快速回显 | 引入局部强制刷新与乐观更新 |
| 统一状态源 | 卡片、详情、统计基于同一任务源 |
| 实时优先 | 事件推送优先，轮询兜底 |
| 串行队列可视化 | 明确显示占用、等待、停滞、可接任务状态 |

### 6. 串行调度与专家扩容策略

由于单 Agent 无法并发，调度中心和协作中枢必须能识别“谁在忙、谁可接、谁阻塞、谁等待”。

因此后续会引入更明确的运行态字段：

| 字段 | 作用 |
|---|---|
| current_task_id | 当前占用任务 |
| queue_length | 当前待处理数量 |
| availability | 可接单 / 忙碌 / 阻塞 |
| last_heartbeat | 最近心跳时间 |
| stale_status | 是否停滞 |
| task_type_tags | 该专家可处理的任务标签 |

扩容方式不再是给单个 Agent 加负担，而是通过增加新的专家 Agent 进行横向分担，例如：

- 代码专家 A / 代码专家 B；
- 测试专家 A / 测试专家 B；
- 检索专家 A / 检索专家 B。

## 四、实施阶段与交付物

### 阶段一：角色硬编码与命名映射审计

目标是找出所有写死旧角色与旧文案的位置，为后续替换与动态化奠定基础。

交付物如下：

| 交付物 | 说明 |
|---|---|
| 角色常量清单 | 前端、脚本、配置、文档中的固定角色引用 |
| 命名映射表 | 旧命名到新命名的完整映射 |
| 风险点清单 | 哪些逻辑与固定角色强耦合 |

### 阶段二：中文命名落地与看板信息架构改造

目标是将所有用户可见界面切换到现代中文表达，并为动态 Agent 做铺垫。

交付物如下：

| 交付物 | 说明 |
|---|---|
| 新版中文文案 | 页面标题、Tab、筛选器、详情、提示文案 |
| 信息架构调整 | 协作中枢、协同讨论、结果报告等模块命名统一 |
| 总控中心能力说明 | 哪些任务可直接处理，哪些必须转交 |

### 阶段三：Agent Registry 与 SOUL 生成机制建设

目标是打通新增 Agent 的规范化入口。

交付物如下：

| 交付物 | 说明 |
|---|---|
| Registry schema | 统一字段规范 |
| Registry 示例 | 至少一份 XX 专家示例 |
| SOUL 生成器 | 根据 Registry 输出完整 SOUL.md |
| 生成模板 | 保证 SOUL 风格稳定可读 |

### 阶段四：自动部署链路建设

目标是将“注册 Agent”升级为“注册即可自动接入”。

交付物如下：

| 交付物 | 说明 |
|---|---|
| 自动部署脚本 | 从规范到目录到配置的完整链路 |
| 配置同步逻辑 | 新 Agent 自动写入必要映射 |
| 接入校验逻辑 | 确认新 Agent 被系统识别 |

### 阶段五：刷新与串行调度优化

目标是提升看板实际使用体验，解决当前的核心体验问题。

交付物如下：

| 交付物 | 说明 |
|---|---|
| 刷新调度器 | 统一轮询与实时更新管理 |
| 局部更新机制 | 任务卡、详情、统计联动刷新 |
| 队列状态展示 | 体现串行占用与等待 |
| 总控中心刷新优先级策略 | 确保总控中心最高实时性 |

### 阶段六：联调与回归验证

目标是验证新增专家、自动生成 SOUL、自动部署和刷新机制是否稳定运行。

交付物如下：

| 交付物 | 说明 |
|---|---|
| 新专家接入验证 | 新增一个专家并完整打通接入 |
| SOUL 生成验证 | 检查完整性、可读性与执行一致性 |
| 看板刷新验证 | 检查操作回显、状态一致性、等待态可视化 |
| 总控中心约束验证 | 验证默认不被长任务占满 |

### 阶段七：文档、版本日志与项目痕迹交付

目标是让后续维护者可以直接接手。

交付物如下：

| 交付物 | 说明 |
|---|---|
| README 更新 | 含版本日志 |
| 迁移说明 | 说明旧角色如何映射到新结构 |
| 使用说明 | 如何新增专家、生成 SOUL、自动部署 |
| 项目痕迹文件 | 记录本次改造的范围与结果 |

## 五、优先级顺序

建议按以下顺序实施：

| 优先级 | 内容 | 原因 |
|---|---|---|
| P0 | 命名映射与角色硬编码排查 | 后续所有改造的基础 |
| P0 | Registry 规范设计 | 扩展与自动化入口 |
| P1 | SOUL 自动生成 | 降低新增 Agent 成本 |
| P1 | 自动部署链路 | 让扩展真正落地 |
| P1 | 刷新调度与串行状态回显 | 解决当前体验核心问题 |
| P2 | UI 细节与动效优化 | 价值高但不应先于架构调整 |
| P2 | 文档与迁移补充 | 功能稳定后统一收尾 |

## 六、验收标准

本次改造完成后，应至少满足以下验收条件：

| 验收项 | 标准 |
|---|---|
| 命名统一 | 看板、文档、主要 SOUL 已切换到现代中文体系 |
| 总控中心实时性 | 默认不被长任务占满，能快速响应新任务 |
| 系统性修复边界 | 仅系统级变更可触发修复模式 |
| 新专家接入 | 新增一个 XX 专家无需手工改多处页面逻辑 |
| SOUL 生成 | 能从规范自动生成完整 SOUL.md |
| 自动部署 | 新 Agent 可标准化自动接入系统 |
| 刷新体验 | 回显明显改善，局部状态更一致 |
| 串行可见性 | 可清晰看到专家占用、等待、停滞状态 |
| 兼容性 | 原有主要工作流仍可运行 |
| 文档完整性 | README、版本日志、项目痕迹完整 |
