# 最终交付检查关卡补充说明

## 背景

本次复盘中补充了一条关键流程约束：**任务在返回“完成”和最终结果之前，不能仅凭执行结束或结果归档就直接交付，必须先经过最终交付检查。**

这一约束用于避免“结果已经产出，但尚未验证是否真正满足原需求”时过早宣告完成。

## 补充后的流程规则

| 阶段 | 规则 |
|---|---|
| 结果归档后 | 必须进入最终交付检查，而不是直接返回完成 |
| 检查通过 | 才允许进入最终交付 |
| 检查不通过 | 返回修复链路继续处理 |
| 存在无法自主决策的分歧 | 返回用户确认，再继续推进 |

## 预期行为

后续在任务执行中，应遵循以下判断顺序：

1. 执行完成并形成结果；
2. 汇总并归档结果；
3. 对照原需求、验收条件和关键输出检查是否满足；
4. 若满足，则返回最终交付；
5. 若不满足，则继续修复；
6. 若修复方向涉及取舍、歧义或超出自主决策边界，则询问用户。

## 结论

这一步补上之后，流程从“执行完成即可交付”修正为“**执行完成 → 归档 → 最终检查 → 通过后交付**”。

这比原来的表达更完整，也更符合实际可靠交付所需要的闭环控制。
