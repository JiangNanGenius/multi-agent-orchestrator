# 归档规则与交付口径对齐结论及 Agent 检查单

> 本文用于在上下文丢失后，重新收敛 **multi-agent-orchestrator_public** 项目中与“归档机制、总控交付口径、路线图承诺边界、Agent 自检清单”相关的结论，并作为后续继续执行看板任务的稳定锚点。

## 一、当前已还原出的真实任务目标

结合项目中的恢复留痕、前后端实现、既有治理文档以及用户补充口径，可以较高置信度判断：此前“看板任务”的真实目标并不是继续追加自动化脚本，而是要**核实真实行为、修正文档口径、统一治理规则，并沉淀一份给 agent 使用的检查单**。[1] [2] [3]

| 主线 | 当前确认的真实目标 |
|---|---|
| 归档机制核查 | 确认是否真的存在“任务完成后静置 N 时间自动归档”的实现 |
| 产品口径收敛 | 将归档说明统一为“通过看板操作或明确指令触发”，而不是依赖模糊的自动时间阈值 |
| 交付链路约束 | 明确任务不能仅因执行结束或进入归档页就直接宣告完成，而必须经过上下文回读与最终交付检查 |
| 治理沉淀方式 | 不继续堆脚本，而是给 agent 提供一份可执行、可复查的检查单 |

## 二、当前代码与文档事实

从当前代码实现看，系统已经支持**单任务归档**与**批量归档已完成任务**，但没有找到“静置多久自动归档”的明确实现证据。后端 `/api/archive-task` 只处理 `taskId + archived` 或 `archiveAllDone` 两种显式归档请求；前端看板也只暴露“归档 / 取消归档”与“一键归档”入口。[4] [5] [6]

| 观察项 | 当前事实 | 判断 |
|---|---|---|
| 单任务归档 | 已实现 | 属于显式归档 |
| 批量归档已完成任务 | 已实现 | 属于显式批量归档 |
| 静置时长自动归档 | 未发现明确实现 | 不能作为对外承诺 |
| 看板一键归档 | 已实现 | 说明归档入口以看板操作为主 |
| 聊天归档专门入口 | 当前仓库未检出明确实现位置 | 若要对外表述，应写为“可通过明确指令流程接入”，不应写成“已存在明确自动阈值机制” |

需要特别指出的是，部分高可见文档仍保留了“任务完成后自动归档”或“自动生成归档记录”这类较强表达，而这与当前代码事实并不完全一致，容易让人误解为系统已经承诺了一个定时静置自动归档机制。[7] [8]

## 三、应采用的统一产品口径

基于当前代码事实与用户新增要求，后续最稳妥的统一口径应收敛如下。

| 主题 | 统一口径 |
|---|---|
| 归档机制 | 归档应以看板显式操作为主，或通过明确的任务/会话指令流程触发；不应对外宣称存在“完成后静置 N 时间自动归档”的既定规则 |
| 总控交付 | 任务完成后，总控应优先返回结果；只有在存在风险、分歧、缺失验收证据或无法自主决策时，才额外提出问题或回到用户确认 |
| 结果归档与交付关系 | 结果归档不是“自动等于完成”，而是交付前的一步；归档后仍需进行上下文回读与最终交付检查 |
| 路线图承诺边界 | 只有“已落地能力”与“已确认自有规划”属于正式承诺；外部启发项不得计入当前完成度 |

上述口径与现有治理文档主线是一致的。项目文档已经把主流程补齐为：**执行完成 → 上下文整理/必要时压缩 → 结果归档 → 回读任务摘要与验收条件 → 最终交付检查 → 通过后交付**。[2] [9] [10]

## 四、此前看板任务最可能尚未收尾的待办

当前最可能仍未完成的，不是“再做一个自动归档脚本”，而是以下三类收口工作。

| 优先级 | 待办 | 目的 |
|---|---|---|
| P0 | 复核并修正文档中所有“自动归档”强表述 | 避免口径与真实实现脱节 |
| P0 | 给出一份对用户可复述的最终判断 | 明确回答“没有发现静置多久自动归档的实现证据” |
| P1 | 在现有检查单基础上补充归档与交付检查项 | 满足“不做脚本、改做 agent 检查单”的要求 |
| P1 | 必要时补项目留痕与 README 对齐 | 防止后续再次丢上下文或重复核查 |

## 五、面向 Agent 的执行检查单

下面这份检查单可直接作为后续部署前、联调前和最终交付前的自检模板使用。它的目标不是替代实现，而是确保 agent 不再沿着错误口径继续扩写。

### 5.1 归档规则检查

| 检查项 | 要求 | 通过标准 |
|---|---|---|
| 归档入口核对 | 核对前端与后端实际入口 | 能指出单任务归档、批量归档入口及其调用链 |
| 自动阈值核对 | 搜索是否存在基于静置时长的自动归档逻辑 | 若未找到，明确写“未发现”，不得自行补脑 |
| 文档措辞核对 | 搜索 README、getting-started、宣传文案中的“自动归档”字样 | 若表述强于实现，应标记为待修正 |
| 聊天归档表述核对 | 检查是否已有明确指令入口实现 | 未落地时只能写成“可作为后续接入方向”，不能写成既有能力 |

### 5.2 最终交付检查

| 检查项 | 要求 | 通过标准 |
|---|---|---|
| 上下文回读 | 回读任务目标、用户新增要求、阶段摘要、风险与验收条件 | 回读证据明确，未遗漏关键新增口径 |
| 结果归档 | 确认任务已有可追溯结果留痕 | 至少能定位归档页、结果报告或留痕文件 |
| 最终 QA | 判断结果是否满足真实需求，而非只看任务是否执行结束 | 不满足则继续修复，不得直接说完成 |
| 用户确认边界 | 存在无法自主判断事项时先询问用户 | 不以假设代替确认 |

### 5.3 路线图与承诺边界检查

| 检查项 | 要求 | 通过标准 |
|---|---|---|
| 路线图分层 | 区分“已落地能力 / 已确认自有规划 / 外部启发候选方向” | 仅前两层作为正式承诺 |
| 完成度判断 | 不把候选方向算作项目欠账 | 结论不被外部灵感条目放大 |
| 用户口径一致性 | 检查路线图描述是否与用户新增口径一致 | 不再把参考项写成已承诺功能 |

## 六、当前阶段性结论

现阶段已经可以给出一个相对稳定的判断：**当前仓库没有提供“任务完成后静置多久不管就自动归档”的明确实现证据；真实行为更接近“任务完成后可在看板中显式归档、可批量归档，并在结果归档页形成留痕”。** 同时，项目治理主线已经补入“上下文整理、结果归档、最终交付检查”的链路，因此后续对外口径不应再把“归档”误写成“自动完成”的同义词。[1] [4] [5] [6] [9] [10]

如果继续推进本轮看板任务，优先顺序应是：**先修正高可见文档中的自动归档强表述，再把本检查单并入现有 AI 检查清单或 README 的治理入口，最后再视需要补充聊天归档能力的真实实现。**

## 七、补充核查结论：后台定时巡检不等于自动归档

本轮又沿着用户截图恢复出的线索，继续核查了项目里是否存在“后台定时任务在任务完成后静置一段时间自动归档”的实现。复核重点覆盖 `dashboard/server.py`、`dashboard/dashboard.html`、`agentorchestrator/frontend/src/components/AgentOrchestratorBoard.tsx` 以及现有恢复文档。核查结果显示：项目**确实存在后台定时巡检线程**，但其职责是停滞任务治理，而不是完成任务自动归档。[11] [12] [13]

在 `dashboard/server.py` 中，服务启动后会额外拉起一个后台线程，每 **120 秒**执行一次 `handle_scheduler_scan(threshold_sec=180)`。该扫描逻辑会根据任务停滞时长与调度配置，执行**自动重试、升级协调、自动回滚或自动挂起**等动作；但在这条链路里，并没有调用 `handle_archive_task`，也没有出现“Done/Cancelled 后超过某时间即归档”的判断条件。[11]

| 核查项 | 当前结论 |
|---|---|
| 是否存在后台定时线程 | 存在 |
| 是否存在周期性扫描 | 存在，每 120 秒一次 |
| 扫描是否处理停滞任务 | 是 |
| 扫描是否自动归档已完成任务 | **未发现** |
| 是否存在静置 N 时间后自动归档阈值 | **未发现** |

这一点也和前端行为相互印证。无论是 `dashboard/dashboard.html` 还是 React 版 `AgentOrchestratorBoard.tsx`，当前都把归档入口落在**单任务归档 / 取消归档**与**一键归档已完成任务**这两类显式操作上；而“🧭 总控巡检”调用的是 `/api/scheduler-scan`，语义上对应巡检与调度治理，不应被理解为归档入口。[12] [13]

> 因此，目前最准确的判断应当是：**仓库存在自动巡检，但不存在已确认的“静置多久后自动归档”机制。自动巡检负责处理停滞任务，不负责把完成任务自动送入归档。**

这意味着，面向用户和面向后续 agent 的统一口径还应进一步收紧。归档目前应描述为**通过看板显式操作触发**；如果未来补齐聊天指令归档，则可以再把“聊天指令归档”纳入正式能力口径，但在当前代码事实下，仍不宜把它描述为既有、稳定、可验证的自动机制。[1] [11] [12] [13]

| 主题 | 本轮补强后的统一口径 |
|---|---|
| 自动化巡检 | 用于停滞任务扫描、重试、升级、回滚与挂起 |
| 自动归档 | 当前**未发现明确实现** |
| 归档触发方式 | 当前以看板显式操作为主 |
| 聊天归档 | 更接近后续产品口径或接入方向，而非当前已核实能力 |
| 完成后的总控行为 | 应以返回结果为主，归档与最终检查属于交付链路的一部分 |

补上这一轮核查后，可以把阶段性最终结论进一步压实为：**当前仓库虽有后台定时巡检机制，但它并不承担自动归档职责；截至本轮检查结束，仍没有发现“任务完成后静置 N 时间自动归档”的代码证据。** 因而，项目对外表述、看板解释、以及后续 agent 自检，都应以“显式归档 + 交付前最终检查”的治理口径为准，而不是延续“完成后会自动归档”的强表述。[11] [12] [13]
