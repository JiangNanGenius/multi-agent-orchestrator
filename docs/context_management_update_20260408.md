# 上下文管理机制补充说明

## 背景

本次复盘中确认，现有流程虽然已经补上了“最终交付检查”关卡，但**仍然缺少一套被明确写出来的上下文管理规则**。这会带来两个问题：其一，长任务在多轮执行后容易因为上下文堆积而丢失关键信息；其二，即使最终做了交付检查，也可能因为中途没有形成稳定的上下文摘要与回读机制，导致交付物与最初需求出现偏移。

因此，后续流程不能只要求“做完再检查”，还必须要求：**在执行过程中持续维护上下文，在接近窗口上限时及时压缩，在压缩后保留可回读的任务摘要，并在最终交付前重新对照这些摘要进行复核。**

## 补充后的上下文管理规则

| 阶段 | 必须动作 | 目的 |
|---|---|---|
| 任务启动前 | 提炼任务目标、约束、验收标准、用户明确偏好，并写成稳定摘要 | 防止一开始就丢失原始需求边界 |
| 执行过程中 | 每完成一个关键阶段，就沉淀一次阶段结论、待办、风险与未决问题 | 防止多轮操作后只剩局部记忆 |
| 上下文接近上限时 | 先压缩历史过程，再保留“当前目标、已完成事项、未完成事项、关键文件、下一步动作”的续写摘要 | 保证后续轮次可无缝接续 |
| 发生分支决策时 | 记录已采用方案、被放弃方案及原因 | 避免后续回读时不知道为何这样做 |
| 最终交付前 | 回读任务目标、用户新增要求、阶段摘要、风险记录与验收条件，再进行最终交付检查 | 防止“过程正确但结果偏题” |
| 无法自主判断时 | 立即向用户确认，而不是在缺失上下文的情况下继续假设 | 防止错误决策放大 |

## 建议维护的最小上下文结构

为了让上下文管理可执行，而不是只停留在口号层，后续建议每个较复杂任务至少维护以下结构。

| 字段 | 含义 |
|---|---|
| 当前目标 | 当前这轮任务真正要完成什么 |
| 用户约束 | 用户明确提出的限制、偏好、禁区 |
| 已完成 | 已经完成并可确认的事项 |
| 未完成 | 仍需继续推进的事项 |
| 风险与分歧 | 尚未解决的问题、可能影响交付的风险 |
| 关键文件 | 当前最重要的文件、图、脚本或输出位置 |
| 下一步动作 | 下一个最合理的动作 |

## 与最终交付检查的关系

上下文管理不是独立于交付检查之外的附属动作，而应该成为交付检查的前置条件。

也就是说，后续流程应从：

**执行完成 → 结果归档 → 最终交付检查 → 交付**

进一步补全为：

**执行完成 → 上下文整理/必要时压缩 → 结果归档 → 回读任务摘要与验收条件 → 最终交付检查 → 通过后交付**

如果在“回读任务摘要与验收条件”这一步发现以下任一情况，就不能直接进入完成状态：

| 情况 | 处理方式 |
|---|---|
| 发现结果偏离原目标 | 回到修复链路 |
| 发现中途新增要求未纳入 | 先补齐，再复核 |
| 发现关键约束未验证 | 先补验证，再决定是否交付 |
| 发现存在取舍但无明确依据 | 询问用户 |

## 预期行为

后续在复杂任务中，应遵循以下顺序：

1. 先固定任务目标、约束与验收标准；
2. 执行过程中持续沉淀阶段摘要；
3. 上下文接近上限时进行压缩，并生成可续写摘要；
4. 每次进入新阶段前，先回读最近稳定摘要；
5. 交付前重新回读原始目标、用户追加要求、阶段结论与风险；
6. 确认满足后，才允许进入最终交付；
7. 若不满足则继续修复；
8. 若无法自主决策则询问用户。

## 结论

补上这一步之后，流程不再只是“做完后检查”，而是升级为：

**开始前固定目标，过程中维护上下文，接近上限时压缩归档，交付前回读复核，确认满足后再交付。**

这会让长任务、多阶段任务和多次需求变更任务的稳定性更高，也更符合真正可持续的多 Agent 协作闭环。
