# 当前架构与处理逻辑总览

> 本文档用于基于当前仓库**实际实现**整理系统的整体架构、模块边界与主要处理链路，优先描述已经落地的运行主线，而不是历史设想或早期叙事。[1][2][3][4][5][6][7]

## 一、系统定位

当前项目可以概括为一套**面向复杂任务治理的多智能体编排系统**。它的目标并不是把多个角色简单拼接成一个聊天入口，而是把任务纳入一条可治理、可观测、可回退、可归档的执行链路中。系统从公开入口上分为**正式版前端层、后端服务层、事件编排层、角色配置层、文档治理层**五个部分；从运行逻辑上则表现为“任务进入系统后，先被结构化为任务对象，再通过状态机和事件总线驱动中心与专家角色协作，最后在正式版界面中持续可见并沉淀为结果归档”的治理闭环。[1][3][4][5][6]

下表概括了当前仓库中各主要层级的职责边界。

| 层级 | 主要目录/入口 | 当前职责 | 是否直接面向用户 |
| --- | --- | --- | --- |
| 文档与治理层 | `README.md`、`docs/current_capability_boundary.md`、`docs/current_architecture_overview.md` | 说明系统边界、部署方式、能力范围与长期治理口径 | 是 |
| 角色配置层 | `agents.json`、`agents/` | 定义中心与专家角色、分组说明、提示词与职责边界 | 间接 |
| 正式版前端层 | `agentorchestrator/frontend/` | 提供任务发布、监控、技能管理、会话、归档、搜索与协作讨论等界面 | 是 |
| API 与应用层 | `agentorchestrator/backend/app/main.py`、`agentorchestrator/backend/app/api/*` | 提供任务、角色、事件、管理与实时连接接口 | 否 |
| 任务领域层 | `agentorchestrator/backend/app/models/task.py`、`services/task_service.py` | 定义任务状态机、持久化结构、状态流转与事件写出 | 否 |
| 事件编排层 | `agentorchestrator/backend/app/workers/orchestrator_worker.py`、event bus / outbox | 消费任务事件、自动推进状态、执行停滞恢复与调度派发 | 否 |

## 二、总体架构

当前系统采用的是一种**“数据库持久化 + Outbox 事件写出 + 事件总线消费 + 前端状态适配”**的分层架构。它不是纯同步请求驱动的单体页面应用，也不是完全去中心化的多智能体自由会话系统。真正的运行骨架在后端：任务首先以数据库记录的形式存在，其次以 outbox 事件的形式被异步投递，再由编排 worker 根据状态与目标角色继续推进。前端更多承担的是**观察、触发、展示与适配**职责，而不是决定任务流转规则。[2][3][4][5][6]

可以把当前架构理解为下面这条主链：

```text
用户/运营者
  ↓
正式版 React 前端
  ↓
FastAPI API 层
  ↓
TaskService（任务服务）
  ↓
PostgreSQL 任务表 + Outbox 事件表
  ↓
Outbox Relay / Event Bus
  ↓
Orchestrator Worker
  ↓
中心角色 / 专家角色
  ↓
任务更新、日志沉淀、归档输出
  ↓
前端轮询/实时接口回显到看板
```

这个设计的关键价值在于，**任务对象是中心事实源，事件是推进机制，前端是可视化外壳**。因此，哪怕某个执行节点暂时失败，系统仍可以依赖数据库中的任务主记录、`flow_log`、`progress_log`、`todos` 与 `scheduler` 等结构恢复上下文，而不会完全依赖即时会话记忆。[3][4][5][6]

## 三、核心模块分层

### 3.1 文档与治理入口

`README.md` 仍然是公开仓库的主入口，它负责统一仓库定位、部署顺序、可见术语、流程图与界面预览，并明确指出当前项目是一个治理型多智能体工程底座，而不是全包式运行时安装包。[1] 与之配套的 `docs/current_capability_boundary.md` 则承担“真实能力边界说明”的角色，用来约束后续恢复上下文时不要把未完成能力误判为已经交付。

这意味着当前项目在治理层面采用的是**“README 负责公开入口，专题文档负责稳定边界”**的策略。对于后续维护而言，这种做法能有效降低因为阶段性日志、临时审计文件或历史术语混入主叙事而造成的理解偏差。

### 3.2 角色配置层

`agents.json` 与 `agents/` 目录共同构成角色配置层。它们负责描述当前有哪些中心角色与专家角色、这些角色如何分组、调用时应遵循什么职责边界，以及对应 SOUL / GLOBAL / 分组文档如何参与运行时提示词装配。当前公开主线已经统一为**center / specialist** 的现代命名口径，旧式代号只允许停留在兼容层或历史说明中，不再作为默认公开叙事。[1][7]

这一层本质上不是界面配置，而是**编排治理的角色语义层**。它决定了“谁可以接收什么任务”“谁负责规划、评审、调度”“哪些角色属于执行团队而不是治理中枢”。

### 3.3 前端与看板层

前端主入口位于 `agentorchestrator/frontend/src/App.tsx`。从当前实现看，它已经不是单一任务面板，而是一个带有认证壳层的 React 应用容器，负责组织多个主面板，包括任务看板、运行监控、角色概览、模型配置、技能管理、会话视图、归档面板、模板面板、AI 搜索引擎、自动化面板、任务详情弹窗与协作讨论界面等。[5]

前端状态适配的关键文件是 `store.ts`。这一层定义了可见流程阶段 `PIPE`、状态到阶段的映射、角色元数据目录 `AGENT_ARCHITECTURE`、角色标签归一化规则、部门颜色与文案规范，以及任务如何被转换为可视化节点、标签、时间、归档状态和动态流程。这意味着前端并不是简单原样展示后端字段，而是承担了**兼容映射与展示语义归一化**职责。[6]

| 前端关键文件 | 当前作用 | 对整体架构的意义 |
| --- | --- | --- |
| `App.tsx` | 组合认证壳层与各业务面板 | 决定用户能看到哪些一级能力入口 |
| `store.ts` | 统一前端状态、轮询、角色映射与流程展示 | 将后端任务结构转换为稳定 UI 语义 |
| `api.ts` | 定义接口调用与类型约束 | 作为前后端契约层 |
| `components/*` | 任务发布、技能管理、AI 搜索引擎、归档等具体交互模块 | 承担用户实际操作入口 |

### 3.4 后端 API 与应用装配层

后端主入口在 `agentorchestrator/backend/app/main.py`。当前应用基于 FastAPI 构建，并在生命周期启动时连接事件总线，在关闭时执行清理。它统一装配了任务、角色、事件、管理与 WebSocket 路由，同时暴露 `/health` 与 `/api` 这类轻量级健康与发现接口。[2]

这一层的职责是**把系统能力以服务接口方式组织起来**。从架构视角看，它本身不承载主要业务规则，而是作为传输与装配入口，把外部请求导向任务服务、角色服务、事件查询或实时连接层。

### 3.5 任务领域层

任务领域层由 `models/task.py` 与 `services/task_service.py` 共同构成。`TaskState` 枚举定义了当前系统真实可用的状态集合，包括 `ControlCenter`、`PlanCenter`、`ReviewCenter`、`Assigned`、`Next`、`Doing`、`Review`、`Done`、`Blocked`、`Cancelled`、`Pending`、`PendingConfirm` 等状态；`STATE_TRANSITIONS` 则给出了合法流转图。[3]

`Task` 模型本身不只是一个标题和状态，它同时承载了看板所需的多数关键字段，包括责任组织、当前进展、阻塞原因、最终输出、归档状态，以及四类关键结构化补充信息：`flow_log`、`progress_log`、`todos` 与 `scheduler`。这说明系统并不是通过零散日志文件来理解任务，而是尽量把治理过程沉淀到任务对象内部。[3]

`TaskService` 则负责把这些任务数据作为可变业务对象进行管理。它承担三类核心能力：第一，创建任务并在同一事务中写出对应 outbox 事件；第二，执行带合法性校验和并发控制的状态流转；第三，补充任务进度、待办与调度元数据，并输出统一的 live status 供前端读取。[4]

### 3.6 事件编排层

真正让系统“自动运转起来”的是 `orchestrator_worker.py`。该 worker 监听 `task.created`、`task.status`、`task.completed`、`task.stalled` 等主题，并将事件按任务维度串行处理，以保证同一任务的推进顺序不被并发打乱。[5]

从实现上看，编排 worker 做了四件关键事情。第一，任务创建后自动派发给总控中心；第二，状态改变后根据新状态自动计算下一个应处理的中心或专家；第三，对 `Assigned` 这类特殊状态区分“调度中心回收”与“专家执行派发”两种路径；第四，针对长时间无推进的任务进行停滞检测、重试、升级与阻塞处理。[5]

这意味着系统的“治理感”并不是只写在文档里，而是真实地体现在自动编排策略中。

## 四、当前主处理逻辑

### 4.1 任务创建

任务通常从任务发布入口进入系统，经前端调用 API 后由 `TaskService.create_task()` 创建数据库记录。在创建瞬间，系统会生成 `trace_id`，补全初始状态、组织、负责人、当前描述以及一条首个 `flow_log` 记录。随后，系统把 `task.created` 事件写入 outbox 表，并与任务记录在同一事务中提交。[4]

这里的关键不是“创建了一条任务”，而是**创建任务与生成事件是原子动作**。这样做能避免“数据库里有任务但事件没发出去”或“事件已发但任务主记录缺失”的不一致问题。[4]

### 4.2 状态流转与治理推进

任务进入系统后，其后续推进并不靠前端自由改字串，而是必须经过 `STATE_TRANSITIONS` 所允许的合法流转。例如总控中心可以进入规划中心，规划中心可以进入评审中心或退回总控中心，评审中心通过后进入 `Assigned`，调度中心再把任务送往执行团队。[3]

`TaskService.transition_state()` 在执行状态变更时会先使用 `SELECT ... FOR UPDATE` 锁住当前任务记录，以确保并发更新不会导致 `flow_log` 丢失或状态错乱。然后它校验新状态是否在允许集合中、更新当前组织与描述，并在同一事务中写入新的 outbox 事件。[4]

这条链路的设计说明当前系统的核心处理原则是：

| 原则 | 实现方式 | 目的 |
| --- | --- | --- |
| 状态不可任意跳转 | `STATE_TRANSITIONS` 限定 | 防止治理链路失序 |
| 并发更新必须串行化 | `SELECT ... FOR UPDATE` | 防止日志与状态丢失 |
| 任务变更必须可追溯 | `flow_log` 持续追加 | 保持完整治理轨迹 |
| 状态推进必须外显为事件 | outbox 写入 | 让编排层可继续自动处理 |

### 4.3 调度派发

当任务进入某个新状态后，编排 worker 会依据 `STATE_AGENT_MAP` 或 `ORG_AGENT_MAP` 判断应当通知哪个中心或专家继续处理。例如 `ControlCenter` 对应总控中心，`PlanCenter` 对应规划中心，`ReviewCenter` 对应评审中心，而 `Assigned` 则需要进一步看是否已经确定 `assignee_org`，从而决定是交给调度中心继续分派，还是直接发往某个专家节点。[3][5]

因此，当前系统中的“调度”其实分为两层。第一层是**状态到角色**的规则调度，由 worker 自动完成；第二层是**调度中心到具体专家**的业务调度，需要结合 `assignee_org`、任务内容与执行目标来决定。也正因为如此，`Assigned` 与 `Doing`、`Next` 之间的关系是整个处理中最关键的一段衔接。

### 4.4 专业执行与汇总

一旦任务被送到执行团队，任务会进入 `Doing` 或 `Next` 等执行相关状态。在这个阶段，系统允许执行日志、备注与子任务结构不断写回任务对象。`progress_log` 用于记录执行过程中的更新，`todos` 用于表达拆分后的执行项，`scheduler` 则记录调度元数据。[3][4]

执行完成后，任务可以进入 `Review` 或直接走向 `Done`。如果需要进一步审核、返修或等待用户确认，则可以通过 `ReviewCenter` 或 `PendingConfirm` 等状态继续回流。这说明当前系统已经具备**执行—汇总—返修—确认—归档**这类完整闭环，而不是线性一次性流水线。[3]

### 4.5 停滞检测与自动恢复

当前处理逻辑中一个很重要但容易被忽略的部分，是停滞治理。`orchestrator_worker.py` 会定期扫描处于 `Doing` / `Next` 等关键状态且长时间没有推进的任务，并发布 `task.stalled` 事件。随后，worker 会按“先重试、再升级、最后阻塞”的策略尝试恢复任务。[5]

升级路径大致遵循“专家执行卡住就回调度中心，调度卡住就回评审中心，评审卡住再回规划中心，规划卡住再回总控中心”的层级递进逻辑。只有当升级路径耗尽，系统才会把任务标记为 `Blocked` 并等待人工介入。[5]

这一机制体现了当前系统与普通任务看板的本质区别：它不仅显示任务，还试图**主动治理任务失败与停滞**。

## 五、前端展示逻辑

当前前端不是把数据库字段直接裸露到用户面前，而是通过 `store.ts` 做了一层较强的展示语义适配。`PIPE` 把后端状态压缩成看板可理解的阶段视图，`AGENT_ARCHITECTURE` 把角色 ID 映射为中文/英文标签、职责说明、层级与颜色，`normalizeAgentId()`、`normalizeDeptLabel()`、`normalizeFlowRemark()` 等函数则负责把兼容字段、旧标签或中英混杂文本统一成当前公开口径。[6]

这意味着“公开界面上看到的流程”与“后端内部状态机”并不是一一裸映射，而是前端在兼容与表达之间做了二次组织。当前这种设计有两个直接收益：一是可以继续保留内部兼容字段而不污染公开界面；二是可以在不改动底层任务模型的情况下持续调整可见文案与阶段呈现。

## 六、数据与事件对象

从数据结构看，当前系统的最小治理单元是 `Task`。一个任务至少包含以下几类信息：标识信息、治理状态、执行归属、可视化字段、过程日志与扩展元数据。[3]

| 数据块 | 典型字段 | 作用 |
| --- | --- | --- |
| 标识信息 | `task_id`、`trace_id`、`title` | 唯一标识与链路追踪 |
| 治理状态 | `state`、`assignee_org`、`org` | 表示当前任务在治理链上的位置 |
| 可视化字段 | `owner`、`now`、`eta`、`block`、`output` | 支撑看板展示 |
| 过程日志 | `flow_log`、`progress_log` | 表示状态变化与执行进展 |
| 执行拆解 | `todos`、`scheduler` | 表示子任务与调度元信息 |
| 扩展上下文 | `meta`、`context_window` | 保存补充上下文与压缩窗口信息 |

与之对应，事件对象则主要承担“驱动编排继续向前”的职责。当前至少存在 `task.created`、状态类事件、完成类事件、派发类事件与停滞类事件。任务是事实，事件是推进，二者共同构成系统的执行主线。[4][5]

## 七、当前能力边界

基于当前实现，系统已经比较清晰地具备以下能力：第一，结构化任务治理；第二，中心—专家分层协作；第三，任务状态机与可追溯流转；第四，前端看板可视化与多面板管理；第五，停滞检测与自动恢复；第六，结果归档与运行期观察。[1][2][3][4][5][6]

但与此同时，也应明确几个边界。当前公开版仍然更像一个**可继续产品化的工程底座**，而不是“所有角色都已完成自治执行”的封闭成品。部分历史兼容字段仍存在于内部映射层；前端对实时性的主要依赖仍以轮询与状态适配为主；角色提示词与运行时环境的最终装配也仍取决于部署侧如何接入现有工作区。[1][6][7]

## 九、结论

综上，当前项目的真实架构可以归纳为：**以任务模型为核心，以状态机为规则，以 outbox + event bus 为推进机制，以 orchestrator worker 为自动治理引擎，以 React 看板为可视化外壳，以角色配置与治理文档为长期稳定边界**。[1][2][3][4][5][6][7]

从处理逻辑看，系统并不是“用户发一句话，多个 Agent 自由讨论后直接输出”，而是“任务先入库、再写事件、再按中心/专家链路推进、再在看板回显、再沉淀为可归档结果”。这正是当前仓库最值得保留的架构特征。
