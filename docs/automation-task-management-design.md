# 自动化任务管理功能设计建议

## 结论

基于当前仓库的实现，**“自动化任务管理”不建议简单挂在“Agent 自动化启用”这个开关下面**，而应该拆成两层能力。

第一层是**任务级自动调度能力**。这一层你们其实已经有基础设施了：现有看板服务里，任务对象已经带有 `_scheduler` 元数据，并且支持 `enabled`、`stallThresholdSec`、`maxRetry`、`retryCount`、`escalationLevel`、`autoRollback`、`snapshot` 等字段；系统还具备自动扫描停滞、自动重试、升级协调、自动回滚和自动挂起的逻辑。这说明现在的自动化，本质上已经是“**任务执行过程中的托管调度器**”，而不是一个完整的“自动化任务管理系统”。

第二层才是应该新增的**自动化任务管理层**。这一层负责管理“哪些任务要自动跑、何时触发、触发后生成什么任务、失败后怎么处理、是否周期执行、是否限定到某个 Agent 或组织”。也就是说，它不应只是一个 UI 开关，而应是一套位于任务之上的“规则 / 作业 / 模板”体系。

换句话说，更合理的结构是：

| 层级 | 作用 | 是否已有基础 |
| :--- | :--- | :--- |
| 自动化任务管理层 | 管理自动化规则、周期任务、触发条件、作业模板 | 基本没有，需要新增 |
| 任务级调度层 | 任务创建后监控停滞、重试、升级、回滚 | 已有明显基础 |
| Agent 执行层 | 具体执行派发、推进状态、写入进度 | 已有 |

## 为什么不建议直接绑在“Agent 自动化启用”下面

如果把这个功能直接做成“启用 Agent 自动化时顺便管理自动化任务”，短期看似简单，但长期会出现职责混乱。

因为“Agent 自动化启用”通常表达的是：**某个任务在执行阶段是否允许系统自动推进**。而“自动化任务管理”表达的是：**系统是否应按规则自动创建、自动派发、自动巡检、自动恢复某类任务**。前者是运行时行为开关，后者是任务生产与编排能力，它们不是同一个概念。

你们当前代码也印证了这一点。`dashboard/server.py` 中的逻辑主要是在已有任务上做调度扫描和容错处理，例如超时重试、升级给评审中心或调度中心、必要时回滚到快照状态。这类机制明显属于“任务生命周期托管”，而不是“自动化任务管理入口”。同时，`agentorchestrator/backend/app/api/tasks.py` 已经提供了 `PUT /tasks/{task_id}/scheduler` 这样的接口，说明任务级调度配置已经具备单任务持久化入口。这意味着最自然的扩展方式，是**先把单任务自动化配置做成正式管理面板，再在其上新增规则级自动化作业系统**。

## 推荐方案

推荐采用**“先补任务级管理，再做规则级自动化”**的两阶段方案。

### 第一阶段：把现有调度器能力产品化

这一阶段先不要急着发明新的任务系统，而是把当前 `_scheduler` 相关能力从“代码里存在”变成“用户可管理”。

建议在任务详情页或管理面板增加一个“自动化”区域，直接对应现有任务调度字段，至少包含以下配置项。

| 配置项 | 含义 | 可直接复用现有能力 |
| :--- | :--- | :--- |
| 启用自动托管 | 是否对该任务启用自动扫描与恢复 | 是 |
| 停滞阈值 | 多久无进展视为停滞 | 是 |
| 最大重试次数 | 停滞后最多自动重试几次 | 是 |
| 自动升级 | 重试失败后是否升级到更高协调层 | 是 |
| 自动回滚 | 是否允许回滚到稳定快照 | 是 |
| 当前快照信息 | 当前稳定状态、保存时间、备注 | 是 |
| 最近自动动作记录 | 最近一次重试、升级、回滚、挂起 | 大部分已有 |

这一阶段的目标不是“做新系统”，而是**把已有调度器显性化、可解释化、可控化**。这样用户先能看懂现在系统自动干了什么，后面再加复杂自动化，认知成本才不会失控。

### 第二阶段：新增“自动化任务管理中心”

当第一阶段稳定后，再新增一个独立入口，例如“自动化任务”或“自动化规则中心”。这个模块不要等同于任务列表，而应管理“自动化作业定义”。

这里建议引入一个新实体，例如：

| 实体 | 说明 |
| :--- | :--- |
| AutomationJob | 一条自动化作业定义，例如“每天巡检卡住超过 30 分钟的任务” |
| AutomationTrigger | 触发条件，例如 cron、事件触发、状态变化、Webhook |
| AutomationTemplate | 生成任务时的模板，例如标题、默认指派组织、优先级、标签 |
| AutomationRun | 每次自动化执行记录，方便审计和排错 |

这样一来，“自动化任务管理”就不再是单个任务的一个布尔值，而是一个真正可运营的系统。

## 最合理的产品结构

如果你问“到底应该放在哪”，我的建议是下面这种结构最顺。

| 页面/入口 | 负责什么 | 适合放什么功能 |
| :--- | :--- | :--- |
| 任务详情页 > 自动化 | 管理当前任务是否自动托管 | 停滞阈值、重试、升级、回滚、快照 |
| 管理面板 > 自动化规则中心 | 管理系统级自动化规则 | 定时创建任务、批量巡检、条件触发、失败通知 |
| Agent 配置页 | 管理 Agent 能力与默认行为 | 默认接单策略、可处理任务类型、并发限制 |

也就是说：

> **任务是否自动被托管**，放在任务维度。
>
> **系统如何自动产生和编排任务**，放在管理中心。
>
> **Agent 本身能做什么**，放在 Agent 配置。

这样职责最清楚，也最容易扩展。

## 一个你现在就能落地的 MVP

如果你现在还不想做太大，我建议先做一个最小可用版本，不要一步到位做复杂规则引擎。

MVP 可以只做三件事。

| MVP 功能 | 说明 | 实现复杂度 |
| :--- | :--- | :--- |
| 任务自动化配置面板 | 编辑单任务 scheduler 配置 | 低 |
| 自动动作日志视图 | 展示系统自动重试/升级/回滚记录 | 低 |
| 手动触发按钮 | 提供“立即扫描 / 重试 / 升级 / 回滚”按钮 | 中 |

做完这三项以后，用户就会明显感觉到：系统的自动化不是黑盒，而是可管理、可观察、可干预的。这个阶段已经很有价值。

之后再加第二批能力：

| 第二批功能 | 说明 |
| :--- | :--- |
| 定时规则 | 例如每 10 分钟巡检特定状态任务 |
| 条件建任务 | 例如某状态进入超过阈值自动派生子任务 |
| 模板化自动派发 | 针对不同组织或 Agent 使用默认模板 |
| 自动化执行历史 | 每次规则运行的结果、失败原因、耗时 |

## 技术实现建议

从你们当前仓库结构看，技术上最顺的做法也应分层推进。

### 1. 先复用现有任务 scheduler 字段

你们后端已有任务级 `scheduler` 更新接口，因此第一阶段尽量不要新建复杂表结构。优先复用现有任务模型中的 `scheduler` 字段，把任务自动化控制先接到前端。

建议前端先补这些能力：

| 前端项 | 建议 |
| :--- | :--- |
| 任务详情抽屉/面板 | 新增“自动化”标签页 |
| 表单字段 | enabled、stallThresholdSec、maxRetry、autoRollback 等 |
| 状态展示 | 最近进展时间、已停滞秒数、当前升级级别 |
| 操作按钮 | 保存配置、立即重试、立即升级、立即回滚 |

### 2. 再新增自动化作业表

第二阶段再新增一组独立模型，而不是继续把所有系统级规则塞进 `scheduler`。

建议至少设计为：

| 表/模型 | 关键字段 |
| :--- | :--- |
| automation_jobs | id、name、enabled、scope、trigger_type、trigger_config、template_config、policy_config |
| automation_runs | id、job_id、status、started_at、finished_at、summary、error |
| automation_targets | job_id、target_type、target_selector |

其中 `trigger_type` 可以先只支持两种：`interval` 和 `event`。不要一开始就做很复杂的表达式系统。

### 3. 调度器与自动化作业之间的关系

这个关系建议明确成：

| 能力 | 责任边界 |
| :--- | :--- |
| AutomationJob | 决定是否创建任务、筛选任务、触发动作 |
| Task Scheduler | 负责单个任务进入执行态后的保活与恢复 |
| Dispatch Engine | 负责把任务派发给具体 Agent / 组织 |

也就是说，**作业系统负责“开始什么”，调度器负责“出了问题怎么办”，派发系统负责“交给谁做”**。

## 我对你这个问题的直接回答

如果用最直接的话回答你刚才那句“是在 agent 自动化启用时去怎么样还是说怎么搞”，我的建议是：

> **不要把“自动化任务管理”简单做成 Agent 自动化启用后的附属功能。**
>
> 正确做法是把它分成两个层次：
>
> 1. 先在任务层把已有自动调度能力做成可管理配置；
> 2. 再在管理层新增独立的“自动化规则/作业中心”，负责定时触发、自动建任务和批量编排。

这样做以后，系统结构会更清楚，而且不会把“Agent 自动执行”“任务自动恢复”“规则自动建任务”三种概念混在一起。

## 建议的实施顺序

| 顺序 | 建议动作 | 价值 |
| :--- | :--- | :--- |
| 1 | 给任务详情页加“自动化”配置面板 | 立即可用，改动最小 |
| 2 | 展示自动重试/升级/回滚日志 | 提高可观察性 |
| 3 | 增加手动触发控制按钮 | 便于运维和测试 |
| 4 | 新建“自动化规则中心” | 才真正具备任务管理能力 |
| 5 | 增加定时触发和模板化建任务 | 扩展成完整自动化系统 |
