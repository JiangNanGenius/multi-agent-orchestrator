## 一、当前状态摘要

结合已恢复的看板任务留痕、归档机制核查结论与用户后续补充附件，可以确认：当前项目并非处于“空白待启动”状态，而是已经完成了相当多的现代化改造前置工作，包括命名迁移、SOUL 提示词链路修复、自动化任务管理面板第一阶段接入，以及 AI 优先部署口径的文档调整。[2] [3]

与此同时，本轮真正尚未收口的部分，主要集中在**治理口径统一**与**高可见文档修正**，而不是继续盲目扩写自动化脚本。尤其是“任务完成后是否会自动归档”这一点，当前代码事实与部分文档表述之间仍存在偏差，需要优先收口。[1] [2]

| 自动化巡检 | 已存在 | 用于停滞任务扫描、重试、升级、回滚与挂起，不等于自动归档 [1] [3] |

## 三、建议的执行顺序

如果现在开始继续推进，而不是重新散开多个战线，建议按下面顺序执行。这个顺序的核心逻辑是：先处理**口径失真风险**，再处理**治理入口统一**，最后才继续做更大范围的能力建设。[1] [2] [3]

| 顺序 | 建议动作 | 原因 |
|---|---|---|
| 1 | 修正文档中的自动归档强表述 | 这是当前最直接的认知风险 |
| 2 | 把归档与交付检查单并入正式文档入口 | 让后续 agent 有统一自检基准 |
| 3 | 输出一份“当前真实能力边界”摘要 | 便于团队内部复述与对外口径一致 |
| 4 | 汇总现代化改造的已完成/未完成清单 | 为后续实施提供清晰边界 |
| 5 | 再继续 Agent Registry、自动部署、刷新机制等中期工作 | 这些属于结构升级，应建立在口径稳定基础上 |

## 五、阶段性结论

当前最重要的不是“继续猜测自动归档时间阈值”，也不是“再补一个自动归档脚本”，而是先把已经核清的事实稳定下来。换言之，**新版 TODO 的首要目标是治理收口，而不是功能扩张**。在这一前提下，后续再推进 Agent Registry、自动部署和刷新机制升级，才不会再次因为口径不一致而返工。[1] [2] [3]

如果要把本轮任务压缩成一句话，那么当前最值得执行的新版 TODO 可以概括为：**先统一归档与交付口径，再整理总进度，再继续现代化改造的结构升级。**
