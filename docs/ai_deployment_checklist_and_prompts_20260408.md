# AI 部署检查清单与标准提示词

> 本文用于补齐多Agent智作中枢在部署前与增量更新阶段的 AI 辅助校验能力，统一环境探测、风险提示、执行前确认、增量验证与异常分流口径。

---

## 一、目标

在准备部署或执行增量更新时，AI 不应直接给出“可以上线”的结论，而应先完成一轮**结构化部署前检查**。这轮检查的目标不是代替执行脚本，而是作为执行前的判断层与交付前的验收层，明确当前环境是否具备运行条件、当前改动是否覆盖关键链路、是否存在需要用户确认的高风险操作，以及是否仍有缺口需要先修复。

这份清单尤其面向以下场景：一是首次安装多Agent智作中枢；二是对前端、看板、同步脚本、SOUL、Registry 或部署脚本做过修改之后的增量更新；三是在正式部署前，要求 AI 根据现有仓库状态给出“可部署 / 不可部署 / 需确认后部署”的结论。

---

## 二、部署前总原则

部署前的 AI 判断必须遵守以下主逻辑：先检查，再决定；先发现风险，再执行；先做最终交付检查，再返回完成。如果检查不通过，应继续修复；如果遇到不能由系统自动决定的取舍，例如是否覆盖已有运行时配置、是否允许影响现有在线任务、是否接受只读建议而非直接写入运行时配置，则必须先询问用户。

| 原则 | 要求 |
|---|---|
| 先检查后执行 | 任何部署建议前都要先做环境与改动面检查 |
| 先验收后交付 | 不能跳过最终交付检查直接返回“已完成” |
| 风险前置暴露 | 高风险操作必须提前说明影响范围 |
| 无法自主决策先询问 | 涉及覆盖、删除、停机、迁移等操作时先问用户 |
| 增量更新优先最小改动 | 优先验证受影响链路，不做无边界重写 |

---

## 三、部署前检查清单

### 3.1 环境探测

部署判断首先检查运行环境是否完整。这里的重点不是只看仓库能否打开，而是确认运行多Agent智作中枢所需的关键入口是否存在，包括 Python、Node.js、OpenClaw 初始化环境、工作区目录、数据目录、前端构建依赖以及脚本运行权限。

| 检查项 | 验收标准 | 失败时处理 |
|---|---|---|
| Python 运行环境 | `python3` 可执行，脚本无语法错误 | 先修复解释器或脚本问题 |
| Node.js / 前端依赖 | 前端可构建，必要依赖可安装 | 修复依赖或锁定版本 |
| OpenClaw 初始化 | 已完成 `openclaw init` 或存在运行时目录 | 提示用户先初始化 |
| 工作区目录 | `~/.openclaw/workspace-*` 可创建或已存在 | 修复权限或补建目录 |
| 数据目录 | `data/` 可写，关键 JSON 可生成 | 修复权限或路径 |
| 脚本权限 | `install.sh`、循环脚本可执行 | 补充执行权限 |

### 3.2 核心能力检查

完成基础环境检查后，应继续检查本项目的四条核心链路是否闭合：**配置同步链路、Registry/SOUL 生成链路、看板展示链路、结果归档与交付链路**。如果其中任何一条只能停留在“文档里有说明”但没有运行证据，就不能直接下“完美完成”的结论。

| 能力链路 | 检查重点 | 通过标准 |
|---|---|---|
| 配置同步 | `scripts/sync_agent_config.py` 可运行并产出最新配置 | 能写出 `data/agent_config.json` 与 Registry 产物 |
| SOUL / Registry | 可根据现有角色产出 spec、快照与部署 sidecar | 至少生成 spec、SOUL 快照与工作区 sidecar |
| 看板展示 | 前端或看板资源可构建，关键面板可读 | 构建通过，关键面板没有显著缺项 |
| 结果归档与交付 | 任务结果可归档，交付前有检查逻辑 | 明确存在“检查不通过则返修”的机制 |

### 3.3 风险提示与执行前确认

AI 在部署前还必须主动给出风险提示，而不是只报“通过/失败”。尤其是以下几类情况，即使技术上可以继续，也必须先提示或先确认。

| 风险类型 | 说明 | 处理方式 |
|---|---|---|
| 覆盖现有运行时配置 | 可能影响用户已有 Agent 配置 | 默认只给建议，不直接覆盖 |
| 重启 Gateway / 服务 | 可能中断当前运行中的任务 | 先提示，再执行 |
| 改写运行时 SOUL | 会影响后续角色行为 | 需要明确说明来源与影响 |
| 增量更新涉及多链路 | 可能同时影响前端、脚本、看板与数据格式 | 执行前列出受影响面 |
| 缺失关键凭证或渠道 | 部署后仍无法真正运行 | 提醒用户补配置，不伪装成“已完成” |

## 四、增量更新标准校验流程

对于前端、看板、同步脚本和服务状态，推荐采用统一的最小闭环校验顺序。这样做的目的是避免每次改完一个点，就凭主观印象判断“应该没问题”。

### 4.1 前端改动

当前端界面、双语文案、看板组件或交互流程被修改后，AI 应先确认变更文件，再执行前端构建检查，最后核对是否有与交互逻辑相关的联动数据字段被破坏。如果只是文案修改，仍然应该至少做一次构建验证；如果涉及组件状态流转，则还应核对接口返回字段与本地状态是否匹配。

### 4.2 看板与数据联动改动

如果改动涉及任务状态、上下文窗口可视化、串行执行状态、实时心跳、归档面板等功能，则不能只看前端构建结果，还要检查数据刷新脚本、任务数据结构、看板读取逻辑与渲染字段是否一致。也就是说，看板功能是否完成，不能只看按钮有没有，而是要看**按钮 → 数据 → 刷新 → 渲染 → 验收**这条链是否完整。

### 4.3 同步脚本与部署链路改动

如果改动 `scripts/sync_agent_config.py`、安装脚本或 Registry 相关逻辑，标准检查顺序应为：先做 Python 语法检查，再执行脚本本身，再确认是否产生新的 `agent_config.json`、Registry specs、SOUL 快照与工作区 sidecar，最后检查是否仍保留运行时配置的安全边界。如果新增专家接入是本轮重点，还应验证新增目录或新增 spec 能否被自动发现并纳入同步结果。

### 4.4 服务状态检查

准备部署前，AI 还应检查当前服务状态是否满足上线条件。这里至少包括：Gateway 是否可用、前端构建产物是否存在、看板服务是否可启动、刷新循环脚本是否具备运行条件，以及部署后是否需要人工执行渠道配置或凭证补齐。

## 五、标准 AI 提示词

以下提示词用于在部署前请 AI 进行一轮统一检查。它不是给用户看的产品文案，而是给执行者或自动化代理使用的工作提示词模板。

### 5.1 通用部署前检查提示词

> 请把当前仓库视为“准备部署前的候选版本”，按以下顺序执行检查：
> 1. 检查运行环境是否完整，包括 Python、Node.js、OpenClaw 初始化、工作区目录与数据目录；
> 2. 检查本轮改动影响的链路，至少覆盖前端、看板、同步脚本、Registry/SOUL、结果归档与最终交付检查逻辑；
> 3. 对每一项给出“已满足 / 部分满足 / 未满足”的判断，并说明证据；
> 4. 若存在高风险操作、覆盖现有配置、重启服务、缺少关键凭证或无法自主决定的取舍，先列出并等待用户确认；
> 5. 只有在关键链路全部通过后，才能输出“可部署”；否则先继续修复或明确列出阻塞项。

### 5.2 增量更新检查提示词

> 请把当前任务视为一次增量更新，而不是首次安装。先识别本轮改动影响的文件和模块，再执行最小必要的验证：前端改动做构建检查；同步脚本改动做语法与实际运行检查；看板数据联动改动检查字段与渲染链；部署相关改动检查产物与运行时边界。最终输出受影响范围、通过项、失败项、仍需用户确认项，不要在未完成最终交付检查前直接宣布完成。

### 5.3 新增专家零手工接入检查提示词

> 请检查当前仓库是否已经支持新增专家的零手工接入闭环。重点验证：新增 `agents/<agent_id>/SOUL.md` 或新增 Registry spec 后，是否能被同步脚本自动发现；是否能自动生成或更新 Registry 产物；是否能部署到对应 workspace；是否能进入看板可见配置；是否仍保持对运行时配置的安全边界。若任何一环仍依赖固定清单或手工映射，请明确指出并继续补强。

## 六、最终交付前复核口径

在部署前的最终复核阶段，AI 不能只说“功能差不多都有了”，而要明确回答下面四个问题：第一，逻辑闭环是否成立；第二，关键能力是否有运行证据；第三，新增改动是否经过最小回归验证；第四，是否仍存在需要用户拍板的风险项。只有这四个问题都得到清晰回答，才可以进入“可部署”结论。

| 复核问题 | 必须回答的内容 |
|---|---|
| 逻辑闭环是否成立 | 任务流、交付检查、上下文管理是否进入主流程 |
| 关键能力是否有证据 | 以脚本运行、构建结果、生成产物、文档留痕为依据 |
| 回归是否做过 | 至少覆盖本轮实际修改到的链路 |
| 是否有待确认风险 | 有则列出并等待用户确认，不得跳过 |

---

## 七、当前建议

结合本项目当前演进方式，后续每次准备部署前，建议都以本文作为统一入口：先跑一轮同步与构建，再按本文检查清单做部署前审计，最后再返回“可部署 / 继续修复 / 需你确认后部署”的结论。这样可以把“感觉上差不多完成了”的判断，收敛为可复查、可交接、可迭代的标准流程。

同时，安装文档与 README 中关于部署、增量更新和只读运行时对接的表述，后续也应继续与本文保持一致，避免文档之间出现“脚本会自动完成一切”和“实际上仍需人工确认关键步骤”之间的口径冲突。
