# Opencode 部署与自检执行手册

> 本文不是给终端用户看的简介，而是给 **AI 执行器**看的操作手册。若你是 opencode、自动化代理或其他具备命令执行能力的 AI，请把本文视为当前仓库中**最高优先级的部署执行说明**。在未与本文冲突的前提下，再参考 `README.md`、`docs/getting-started.md` 与 `install.sh`。[1] [2] [3]

## 一、目标

本文的目标不是让 AI “尽量部署”，而是让 AI 以**更高成功率**完成一次正式部署：先确认环境，再执行最小必要动作，再完成正式前端与后端同源链路验证，最后给出有证据的部署结果。当前仓库的有效主链路已经收口为：先完成 `openclaw init`，再执行 `./install.sh`，然后通过 `./agentorchestrator.sh start` 启动正式服务，并直接访问 `http://127.0.0.1:38000/` 使用后端同源托管的正式前端。[2] [3] [4]

## 二、执行原则

AI 在执行过程中必须把“可部署”理解为**有证据的可部署**，而不是“脚本似乎跑完了”。尤其要避免三类常见误判：第一，只装了后端依赖却没有完成正式前端构建；第二，把开发联调入口误当成正式入口；第三，在已有运行时环境上额外安装并拉起不必要的服务，导致口径分裂或状态更复杂。[2] [3] [4]

| 原则 | 执行要求 |
| --- | --- |
| 先检查后执行 | 先确认 OpenClaw、Python、Node.js、仓库路径和初始化状态，再决定是否安装 |
| 最小必要改动 | 只执行当前主链路需要的命令，不自行引入 Docker、额外守护进程或开发模式 |
| 正式入口优先 | 只把 `http://127.0.0.1:38000/` 视为成功入口，不把开发端口当作完成证据 |
| 失败必须止损 | 若前端未构建、注册未补齐或健康检查失败，不得宣布完成 |
| 敏感信息不外显 | 可检查是否缺少密钥，但不得打印密钥值、令牌值或密码文件细节 |

## 三、部署前判定

在真正执行安装前，AI 应先完成一次极小但关键的预检。预检的作用是避免把“环境没准备好”误判为“仓库有 bug”。只要其中任一关键前提不满足，就应先修复前提，再继续部署。[1] [2] [3]

| 检查项 | 通过标准 | 不通过时动作 |
| --- | --- | --- |
| OpenClaw 可执行 | `openclaw` 命令可用 | 停止并提示先安装 OpenClaw |
| OpenClaw 已初始化 | 已存在运行时初始化结果，可进入安装脚本主流程 | 先执行 `openclaw init` |
| Python 可用 | `python3` 可执行 | 停止并修复 Python 环境 |
| Node.js / pnpm 可用 | 前端具备构建条件 | 停止并补齐 Node.js 18+ 与 `pnpm` |
| 仓库工作区存在 | 已进入仓库根目录 | 先切到正确目录 |
| 写权限正常 | 可写 `data/`、可生成构建产物 | 停止并修复权限 |

建议按以下顺序执行预检命令：

```bash
pwd
python3 --version
node --version
pnpm --version
openclaw --version
test -f "$HOME/.openclaw/openclaw.json" && echo OPENCLAW_INIT_OK || echo OPENCLAW_INIT_MISSING
```

若最后一项输出 `OPENCLAW_INIT_MISSING`，应执行：

```bash
openclaw init
```

## 四、标准部署步骤

完成预检后，AI 应按以下顺序执行，不要跳步，不要并行引入额外部署分支，也不要把旧文档中的开发联调步骤混进正式部署流程。[2] [3] [4]

### 第 1 步：进入仓库并确保脚本可执行

```bash
cd /path/to/multi-agent-orchestrator
chmod +x install.sh agentorchestrator.sh
```

### 第 2 步：执行正式安装脚本

```bash
./install.sh
```

安装完成后，不应立刻宣布成功，而应检查三类结果：其一，后端依赖是否安装完成；其二，正式前端构建产物是否存在；其三，运行时 Agent 注册是否已被补齐并通过校验。[2] [3]

### 第 3 步：启动正式服务

```bash
./agentorchestrator.sh start
```

默认不应启用 outbox。只有在用户明确要求高级场景时，才允许使用额外环境变量启动附加链路。[2]

### 第 4 步：执行正式自检

```bash
./agentorchestrator.sh status
curl -fsS http://127.0.0.1:38000/health
curl -fsS http://127.0.0.1:38000/ | head
```

若根路径返回的是正式前端页面内容，而不是空白、错误页或仅有后端报文，才说明“后端同源托管正式前端”这条链路已经成立。[2] [3] [4]

## 五、正式自检矩阵

AI 必须逐项对照下面的自检表，不要只因为安装脚本退出码为 0 就提前结束。只要任一关键项失败，就不能给出“部署完成”的结论。[2] [3] [4]

| 自检项 | 检查方法 | 通过标准 |
| --- | --- | --- |
| OpenClaw 初始化 | 检查运行时初始化结果是否存在 | 已满足安装脚本前提 |
| 安装脚本执行 | `./install.sh` 成功结束 | 无中途终止、无前置校验失败 |
| 正式前端已构建 | 检查正式前端产物存在 | 不出现“只部署后端未部署前端” |
| 服务栈启动 | `./agentorchestrator.sh status` | API、orchestrator、dispatch 处于运行状态 |
| 健康检查通过 | `curl /health` | 返回成功响应 |
| 正式首页可访问 | `curl /` 或浏览器访问 | 根路径能返回正式前端页面 |
| 运行时注册有效 | 安装脚本与同步结果无注册失败 | 不存在关键 Agent 缺失 |
| 敏感信息未外显 | 检查日志与输出内容 | 未打印密钥、令牌、密码值 |

## 六、遇到问题时的分流处理

如果部署失败，AI 不应笼统地说“安装失败”，而应按故障类别继续处理。这样能显著提高下一次修复成功率。[2] [3]

| 现象 | 优先判断 | 处理方式 |
| --- | --- | --- |
| `install.sh` 无法继续 | 多半是前置条件缺失 | 先检查 `openclaw init`、Node.js、权限和脚本输出 |
| 后端起来了但页面打不开 | 多半是正式前端未构建或未托管 | 重新检查安装脚本中的前端构建结果与根路径返回内容 |
| Agent 报缺少密钥 | 多半是运行时模型认证未配置 | 补齐合法认证配置后重新执行安装同步 |
| 状态显示未启动 | 多半是服务栈未完整拉起 | 重新执行 `./agentorchestrator.sh start` 并查看状态 |
| 想额外启用其他服务 | 不是默认路径 | 除非用户明确要求，否则不要自行添加 |

## 七、给 opencode 的推荐执行话术

若由 opencode 执行，推荐直接把下面这段作为工作指令输入，并要求它严格按顺序执行、逐项自检、不要自行发明额外部署路径：

> 你现在要部署 `multi-agent-orchestrator`。请先阅读 `docs/opencode-ai-deployment-runbook.md`，并把它视为最高优先级执行说明。严格按“预检 → `openclaw init` 前提确认 → `./install.sh` → `./agentorchestrator.sh start` → `/health` 与 `/` 自检”的顺序执行。不要使用开发模式，不要引入 Docker，不要额外安装非必要服务。只有在正式前端由后端同源托管且健康检查通过后，才能宣布部署成功。若失败，请按故障类别继续修复并给出证据。

## 八、完成标准

只有同时满足以下条件，AI 才可以输出“本次部署已基本成功”：一是安装脚本完成且无关键前置失败；二是正式服务栈已启动；三是 `http://127.0.0.1:38000/health` 可用；四是 `http://127.0.0.1:38000/` 能返回正式前端页面；五是没有为了“先跑起来”而额外引入主链路之外的部署方式。[2] [3] [4]
