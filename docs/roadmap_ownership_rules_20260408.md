# 路线图归属与承诺口径规则

> 本文用于修正多Agent智作中枢路线图的表达边界，避免把外部项目或参考体系中的能力方向，直接写成本项目已经确认承诺的正式计划。

## 一、核心问题

当前路线图同时混合了三类内容：已经落地的能力、团队已确认要推进的自有规划，以及来自“三省六部”等外部体系的参考方向。如果继续混写，会导致项目边界、完成度评估与部署前预期都被放大或混淆。

## 二、三层结构

| 层级 | 定义 | 是否属于本项目正式承诺 |
|---|---|---|
| 已落地能力 | 仓库中已有实现痕迹并经过基本验证的能力 | 是 |
| 已确认自有规划 | 团队明确决定推进、但尚未全部完成的路线 | 是 |
| 外部启发候选方向 | 参考外部体系得到的想法，尚未正式立项吸收 | 否 |

## 三、判定规则

只有前两层属于本项目正式表达范围。第三层只能作为候选方向，不能参与当前版本完成度判断，也不能作为部署前必须满足的上线条件。

## 四、重写 ROADMAP 的原则

`ROADMAP.md` 后续应拆成三个区块：第一部分写已落地能力；第二部分写已确认自有规划；第三部分单列“外部启发候选方向”，并明确注明其参考属性，不构成当前版本承诺。

## 五、对完成度判断的影响

以后讨论“项目是否已经完成”时，只能对照已落地能力与已确认自有规划，不能把所有参考性条目都算进去。否则会把外部灵感误当作本项目欠账。

## 六、执行建议

下一步应重写 `ROADMAP.md` 的 Phase 2 与 Phase 3，把真正属于本项目的规划保留在正式路线中，把借鉴自外部体系但尚未正式采纳的方向迁移到候选区，并明确其来源属性与非承诺性质。
