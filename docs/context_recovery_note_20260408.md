# 看板任务上下文找回记录

## 一、当前已基本还原出的任务背景

本轮“看板任务”并不是一个全新的零散问题，而是在 **multi-agent-orchestrator_public** 项目上继续做一次产品口径与实现对齐核查。结合项目文档、代码实现、既有交付记录与用户补充截图，目前可以较高置信度判断，此前任务的真实目标主要集中在以下四条主线。

| 主线 | 当前理解 |
|---|---|
| 归档机制核查 | 确认“完成后多久不管就自动归档”是否真实存在，以及具体触发条件在哪里 |
| 产品口径修正 | 将口径收敛为“归档应通过看板操作或与 agent 聊天指令触发，而不是依赖不明确的自动定时阈值” |
| 交付链路约束 | 强调任务完成后不能只因为执行结束就宣告完成，而要经过结果归档、上下文回读与最终交付检查 |
| Agent 检查单沉淀 | 不写额外脚本，而是给 agent 留一份可执行检查单，用于后续部署前、联调前与交付前自检 |

## 二、用户新增口径（来自截图）

用户截图中已经明确补充了新的判断标准，这部分应视为当前任务最重要的产品口径输入。

| 用户口径 | 当前含义 |
|---|---|
| “完成已确认自有规划” | 路线图只应对已落地能力和已确认自有规划负责 |
| “对于归档应该通过看板或者跟 agent 聊天指令归档” | 不应再对外声称存在明确的‘完成后静置多久自动归档’规则，归档入口应以手动触发为主 |
| “任务完成后总控就应该返回结果，不提出意见就是基本没问题” | 结果交付应以总控返回结果为主，除非存在明确分歧、风险或验收缺口，否则不应额外制造无意义意见环节 |
| “完成 2 3。4 的意见是不做脚本而是给 agent 做一份检查单” | 本轮更适合产出检查单与治理文档，而不是继续堆自动化脚本 |

## 三、目前已确认的代码与文档事实

### 1. 自动归档阈值目前没有被明确实现

在 `dashboard/server.py` 中，`handle_archive_task` 的行为是：

| 行为 | 现状 |
|---|---|
| 单任务归档 | 接收 `taskId` + `archived`，显式设置归档状态 |
| 批量归档 | `archiveAllDone` 为真时，将所有 `Done` / `Cancelled` 且未归档任务批量归档 |
| 定时阈值归档 | **未发现** |
| 无人操作自动归档时间配置 | **未发现** |

这说明当前代码层面更接近“显式归档”而不是“达到静置时长后自动归档”。

### 2. 看板前端也以手动归档为主

在 `dashboard/dashboard.html` 与 `edict/frontend/src/components/EdictBoard.tsx` 中，均存在以下入口：

| 入口 | 现状 |
|---|---|
| 单卡片归档 | 已提供“归档 / 取消归档”按钮 |
| 批量归档已完成任务 | 已提供“📦 一键归档”按钮 |
| 总控巡检 | 已提供“🧭 总控巡检”按钮，但其语义是巡检，不等于自动归档 |
| 基于静置时长自动归档 | **未发现明确前端策略或提示** |

因此，当前真实产品行为与用户新口径是一致的：**归档入口是看板操作，不是一个对外明确承诺的自动时间规则**。

### 3. README / 流程文档强调结果归档与最终交付检查

`README.md`、`docs/delivery_gate_update_20260408.md`、`docs/context_management_update_20260408.md` 已明确补入以下流程：

**执行完成 → 上下文整理/必要时压缩 → 结果归档 → 回读任务摘要与验收条件 → 最终交付检查 → 通过后交付**

这说明当前文档主线已经不支持“做完立即结束”的粗放叙事，而是强调：

| 环节 | 当前文档口径 |
|---|---|
| 结果归档 | 是交付前必经步骤之一 |
| 最终交付检查 | 是归档之后、正式交付之前的必要关卡 |
| 无法自主判断时 | 需要回到用户确认 |
| 长任务处理 | 需要上下文压缩归档与续写摘要 |

### 4. 路线图口径已被重新约束

`docs/roadmap_ownership_rules_20260408.md` 已明确：路线图应拆为“已落地能力 / 已确认自有规划 / 外部启发候选方向”三层，且**只有前两层属于正式承诺**。这与用户截图中的要求一致。

## 四、当前最重要的差异点

虽然文档主线在逐步收敛，但仍有一个需要在后续结论中明确指出的差异：部分材料里仍把“任务完成后自动归档”写成较强表述，而实际代码更像“完成后可在归档页形成归档记录，显式归档入口在看板或接口中触发”。

| 项目 | 当前状态 | 风险 |
|---|---|---|
| 代码实现 | 以显式归档和批量归档为主 | 真实行为较清晰 |
| 文档个别表述 | 存在“任务完成后自动生成归档记录”“自动归档”类说法 | 可能让人误解为存在定时静置自动归档机制 |
| 用户新口径 | 倾向“看板或聊天指令归档” | 需要作为后续统一口径的优先依据 |

## 五、对此前任务目标的当前推断

如果把此前看板任务压缩成一句话，当前最可能的真实目标是：

> **核实归档机制的真实实现，修正文档中可能引发误解的“自动归档”表述，统一到“通过看板或聊天指令归档；结果交付前必须经过上下文回读与最终交付检查；路线图只承诺已落地能力与已确认自有规划”的治理口径。**

## 六、下一步建议

后续继续执行本任务时，优先顺序建议如下。

| 优先级 | 下一步动作 | 目的 |
|---|---|---|
| P0 | 全面复核 README、getting-started、看板文案中所有“自动归档”表述 | 找出与真实实现不一致的说法 |

| P1 | 必要时补项目留痕文档 | 防止后续再次重复找上下文 |

## 七、当前判断

截至本记录写入时，可以先给出一个阶段性结论：

> **目前没有找到“任务完成后静置 N 时间自动归档”的明确实现证据；现有实现更接近“任务完成后可在看板中手动归档、可批量归档、并在结果归档页形成归档留痕”。**

该结论后续仍需继续对剩余高可见文档与可能的调度代码进行补扫，但已经足以作为当前任务恢复上下文的核心锚点。
