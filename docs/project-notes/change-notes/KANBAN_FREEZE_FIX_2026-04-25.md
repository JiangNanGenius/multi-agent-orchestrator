# 看板卡死修复记录

## 一、问题现象

本轮补入的是一处与协作讨论结束流程相关的稳定性修复。现象上表现为：**看板后端进程仍然存活，但健康检查、首页根路径或实时状态接口偶发长时间无响应**，用户会误以为“看板崩溃了”。从用户提供的现场排查记录看，问题出现时 `uvicorn` 进程仍在，但 `conclude` / `destroy` 相关调用之后服务可能出现明显阻塞。

## 二、根因判断

根因不在于服务进程消失，而在于 `agentorchestrator/backend/app/api/compat.py` 中协作讨论结束链路使用了**同步的 `subprocess.run(...)`** 去逐个调用 OpenClaw Agent 执行 `/new`。这会把请求线程长时间卡在外部子进程上；当参与 Agent 数量较多、单次调用耗时较长时，接口会被拖慢，看板前端也就表现为卡死或无响应。

## 三、修复策略

本轮没有直接照搬用户提供的临时编辑片段，而是做了更稳妥的手工收口，避免把重复函数定义、缩进损坏或临时调试残留带回仓库。最终方案包含两部分：一是抽出 `_collect_collab_discuss_agent_ids(...)` 统一收集参会 Agent；二是新增 `_clear_collab_sessions_async(...)`，把 `/new` 清理动作放到后台守护线程执行，使 `conclude` 与 `destroy` 请求可以立即返回。

| 修复点 | 调整内容 | 目的 |
| --- | --- | --- |
| 参会者提取 | 新增 `_collect_collab_discuss_agent_ids(...)` | 复用会话读取逻辑，避免两处接口各写一套 |
| 会话清理 | 新增 `_clear_collab_sessions_async(...)` | 将阻塞式子进程调用移出主请求路径 |
| conclude 接口 | 写入 `completed` 后立即返回 | 避免结束讨论时拖住后端 |
| destroy 接口 | 删除会话文件后后台清理 Agent 上下文 | 避免销毁讨论时再次卡死 |
| 返回结果 | 增加 `clear_mode: background` | 明确当前清理是后台执行模式 |

## 四、验证结果

本轮已对修改后的 `compat.py` 执行 Python 语法检查，未发现语法错误。同时对当前修复文件做了一轮定向敏感信息复查，命中的 `secrets`、`token`、`password` 均属于项目既有认证字段与标准变量名，没有新增硬编码密钥、API key、私钥或 Bearer 凭据。本轮新增逻辑也没有把认证文件路径、外部令牌或运行时私密配置写入文档或返回结构。

## 五、结论

当前仓库已将这次“看板像是崩溃但实为阻塞”的问题收口为**非阻塞后台清理**方案。后续如果协作讨论参与者更多、单个 Agent 响应时间更长，请优先排查是否仍有其他接口在请求路径中直接执行耗时子进程，而不要只看进程是否还活着。
