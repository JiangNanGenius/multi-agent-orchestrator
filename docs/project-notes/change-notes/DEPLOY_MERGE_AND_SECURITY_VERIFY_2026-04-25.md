# 部署副本合并、脚本升级与敏感信息核查记录

## 一、处理背景

本轮处理目标并不是简单覆盖仓库，而是先审查用户提供的已部署运行副本 `orchestrator-20260425-0823.tar.gz`，识别其中真正属于**源码层优化**的改动，再将这些改动有选择地合并回 `multi-agent-orchestrator` 仓库，同时对部署脚本做进一步收口，避免再次出现“后端部署了但前端没有正式部署”“前后端对接不一致”“启动默认带上不必要进程”等问题。

## 二、上传副本中识别到的有效源码改动

经对比，上传副本中真正需要回收的改动主要集中在后端 API、worker、前端界面与启动脚本，而不是运行时数据目录本身。

| 分类 | 已合并内容 | 说明 |
| --- | --- | --- |
| 后端入口 | `agentorchestrator/backend/app/main.py` | 增加正式前端静态资源挂载与根路径返回 `index.html` 的能力，使正式界面可由后端同源托管。 |
| 兼容接口 | `agentorchestrator/backend/app/api/compat.py` | 合并了记忆中心直读 OpenClaw 工作区、直接搜索接口、登录后改用户名/改密码等能力。 |
| 任务与管理接口 | `admin.py`、`tasks.py` | 收敛了日志读取参数处理，并修复删除任务接口返回结构。 |
| Worker 流转 | `dispatch_worker.py`、`orchestrator_worker.py` | 合并了自动推进状态、任务完成后回总控中心汇报等处理逻辑。 |
| 前端主界面 | `App.tsx`、`AgentOrchestratorBoard.tsx`、`index.css` 等 | 合并了会议界面命名调整、任务面板文案优化、搜索面板重构、记忆中心重构等 UI 与交互更新。 |
| 前端依赖 | `package.json`、`package-lock.json` | 增加 `marked` 依赖，用于搜索结果等 Markdown 内容渲染。 |
| 统一启动脚本 | `agentorchestrator.sh` | 合并了优先使用项目 `.venv` 的能力，并在此基础上继续收口默认进程。 |

## 三、本轮额外做的部署脚本升级

除了吸收上传副本中的已有改动，本轮还对仓库脚本做了进一步升级，以直接对应用户遇到的部署问题。

| 问题 | 本轮修复 | 结果 |
| --- | --- | --- |
| 后端起了但前端没正式部署 | 在后端入口挂载 `frontend/dist`，并在 `install.sh` 中强制校验 `dist/index.html` 是否存在 | 正式访问入口统一为 `http://127.0.0.1:38000/` |
| 前后端环境容易错位 | `install.sh` 统一创建并使用项目内 `.venv`；`agentorchestrator.sh` 也优先使用 `.venv/bin/python3` | 安装与启动使用同一套 Python 运行环境 |
| 构建阶段容易“假成功” | `install.sh` 中当没有 Node 且没有现成 `dist` 时直接失败，而不是误报安装完成 | 避免只完成后端准备却没有正式前端 |
| 依赖安装方式不稳定 | 前端优先使用锁文件安装，`pnpm-lock.yaml` 走 `pnpm`，`package-lock.json` 走 `npm ci` | 降低环境漂移和联调不一致 |
| 默认拉起非必要进程 | `agentorchestrator.sh` 默认仅启动 API、orchestrator、dispatch；`outbox relay` 改为显式启用 | 默认部署链路更精简 |
| 文档口径分裂 | 同步更新 `docs/getting-started.md`、`docs/remote-skills-quickstart.md`、`docs/screenshots/README.md` | 文档与脚本行为保持一致 |

## 四、明确未直接回收的内容

上传副本中还包含大量**部署态或运行态文件**。这些内容已经审查，但未直接并入仓库主线，以避免把环境噪声或敏感状态带回 Git。

| 未并入内容 | 原因 |
| --- | --- |
| `data/` 下的任务、会话、认证状态文件 | 属于运行时数据，不应作为源码提交 |
| `.pids/`、任务工作区、归档目录 | 属于本地运行现场，不应进入主仓库 |
| `registry/specs/*.json` 中多数仅反映部署期模型/时间戳/工作区路径的漂移 | 多为生成态元数据，不应把部署机特征直接写回仓库 |
| 上传副本中新出现的 `main.json` / `main.SOUL.md` 生成物 | 当前仓库并无对应稳定源码入口，判定为部署期生成残留，暂不并入 |

## 五、敏感信息核查结论

本轮先对上传副本中的**拟合并源码文件**执行定向敏感信息扫描，再对当前仓库源码层做一次扩展扫描。结论如下。

> 在本轮拟合并的源码、脚本、前端组件与注册规格中，未发现可直接判定为硬编码 API Key、私钥、Bearer Token 或密码明文的内容。

扩展扫描命中的若干字符串主要属于两类：第一类是变量名或占位字段，例如 `access_token`、`secret_key`；第二类是历史文档中对“生产环境必须替换占位值”的说明。这些命中不构成“已把真实密钥提交到仓库”的证据，但仍提示后续上线时应继续遵循环境变量注入与运行时配置分离原则。

## 七、结论

本轮不是“原样照搬部署目录”，而是完成了**源码回收、部署脚本升级、默认链路收口、正式前端同源托管、敏感信息排查**五项动作。后续只要使用更新后的 `install.sh` 与 `./agentorchestrator.sh start`，正式部署链路将以单一入口工作：后端负责 API，也负责托管正式前端；开发模式仍然保留，但不再与正式部署路径混淆。
