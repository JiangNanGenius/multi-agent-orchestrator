# 当前真实能力边界与治理入口

本文用于作为当前公开仓库的**长期治理入口**。它不记录一次性排查过程，也不复述阶段性上下文恢复细节，而是只保留后续持续维护时最容易被误判的三类事实：一是当前仓库已经稳定具备哪些能力，二是归档、交付与自动化巡检之间的边界到底是什么，三是现代化改造目前哪些部分已经完成、哪些仍在推进、哪些尚未启动。

## 一、当前真实能力边界

当前仓库已经具备一个可运行的多智能体协作编排骨架，主线能力集中在**任务进入治理链路、状态可视化、角色调度、结果沉淀与公开部署说明**几个方面。换言之，它已经不是单纯的概念样例，而是一套可以继续部署、继续改造、并继续收口公开口径的工程底座。

| 维度 | 当前真实状态 | 维护口径 |
| --- | --- | --- |
| 任务治理主链路 | 已具备任务入口、总控预处理、规划、评审、调度、专业执行与结果沉淀主线 | 对外可表述为“可治理、可观测、可审查、可沉淀” |
| 看板与前端 | 已具备看板、任务详情、技能管理、模板中心、全网搜索、自动化中心等主要入口 | 对外可表述为“可运行的操作台”，不应表述为“全部产品化已完成” |
| 命名体系 | 已完成 center / specialist 主命名迁移，公开界面默认不再使用旧组织术语 | 历史兼容值仅允许留在内部映射层 |
| 部署入口 | 已形成 README 主入口、安装脚本、增量同步与前端构建路径 | 推荐口径是“先由 AI 审视环境，再选择最小执行路径” |
| 自动化能力 | 已具备停滞治理、自动动作日志、自动化配置与自动化中心初版 | 不应夸大为“全自动自治系统” |
| 归档能力 | 已支持显式归档与对已完成/已取消任务的批量归档 | 不应表述为“任务完成后静置一段时间自动归档” |
| 公开发布状态 | 已完成主要脱敏、README 预览图补充与公开界面口径收口 | 仍需持续维护文档一致性与阶段性文件归档 |

## 二、归档、交付与自动化巡检的边界

本项目最容易被误判的点，不在于“有没有自动化”，而在于**自动化究竟负责什么**。当前仓库已经存在后台巡检、自动动作日志、停滞阈值、自动回滚等治理能力，但这些能力服务的重点是**停滞任务治理与运行托管**，而不是把任务在完成后静置一段时间自动塞入归档区。

> 当前更准确的对外说法应当是：**系统存在自动化巡检与托管能力，但归档仍以显式归档或已完成/已取消任务的批量归档为主；自动化巡检不等于自动归档。**

| 概念 | 当前真实含义 | 不应再使用的误导性说法 |
| --- | --- | --- |
| 归档 | 将结果、状态与历史沉淀为可追溯记录，入口为显式操作或批量归档 | “任务完成后会自动归档” |
| 最终交付 | 在结果返回用户前做最后检查、必要时返修或请求确认 | “归档就等于交付完成” |
| 自动化巡检 | 定时检查停滞、重试、升级、回滚与运行托管状态 | “巡检线程会自动帮你完成归档” |
| 自动化中心 | 集中展示任务级自动化配置、覆盖率与近期自动动作 | “已经具备完整跨任务规则编排系统” |

因此，后续 README、安装文档、治理说明与对话恢复都应遵循同一条边界：**先返回结果，再做最终检查；归档是显式治理动作；巡检是运行托管机制。** 三者之间有关联，但不是同义词。

## 三、当前现代化改造总进度视图

为了避免后续上下文恢复再次把“历史规划”“阶段性检查单”“已经完成的改造”混在一起，当前总进度只分为三类：**已完成**、**进行中**、**待启动**。

| 状态 | 模块 | 当前判断 |
| --- | --- | --- |
| 已完成 | 命名体系主线重构 | `agents.json`、`agents/` 组文档、主调度脚本、测试主链与公开界面主口径已完成迁移 |
| 已完成 | 三大模块后台任务化收口 | 任务发布、技能管理、全网搜索已统一到“发起后台任务 → 运行中可见 → 完成后沉淀结果历史”主交互 |
| 已完成 | 前端构建产物治理 | `edict/frontend/dist/` 已不再纳入版本控制，避免构建产物不一致继续阻塞提交 |
| 已完成 | README 主入口与预览图 | 主 README 已补入流程图与运行态预览，公开仓库主入口已成型 |
| 已完成 | Todo 合并收口 | 多份 Todo 已合并为根目录 `todo.md`，临时 Todo 已转为归档说明 |
| 进行中 | 文档治理统一 | 仍需继续把阶段性检查单中的稳定结论并入长期入口，并减少零散文档分散表达 |
| 进行中 | 旧审计/阶段性文件归档 | 仍需把历史审计、阶段性 TODO、临时评审材料集中移入独立归档目录，避免混在主线根目录 |
| 进行中 | 前端体验补充回归 | 仍需继续做高中文模块口径复核、兼容内部 ID 可见性检查与构建联调回归 |
| 待启动 | 人工审批闭环 | 仍属于后续正式规划，当前尚未形成完整审批面板与审批记录链路 |
| 待启动 | Agent 绩效评估体系 | 仍未形成稳定的完成率、返工率与趋势分析体系 |
| 待启动 | 知识沉淀与引用溯源系统化 | 仍缺少可检索的经验沉淀、相似任务复用与引用溯源闭环 |
| 待启动 | 部署校验脚本化固化 | 当前主要靠 README 与检查清单，尚未完全落成脚本化入口 |

## 四、长期维护时应优先查看哪些文件

后续如果还要继续推进治理收口，不应再先翻阶段性检查单，而应优先从下面这几类长期入口开始判断项目状态。这样做可以减少把历史审计结论误认成当前待办主线的概率。

| 文件 | 用途 | 维护建议 |
| --- | --- | --- |
| `README.md` | 公开主入口与部署主说明 | 保持对外叙事、流程图、预览图和部署边界一致 |
| `ROADMAP.md` | 已落地能力 / 已确认规划 / 候选方向总视图 | 只记录正式范围，不把阶段性检查单混入路线图 |
| `todo.md` | 当前统一待办主清单 | 新增事项只在这里增量维护 |
| `docs/current_capability_boundary.md` | 当前真实能力边界与治理口径稳定摘要 | 后续所有口径争议先回到本文校对 |
| `docs/project-progress-log.md` | 阶段性交付与改造留痕 | 用于看版本推进脉络，但不替代主 Todo |
| `edict/WORKLOG_2026-04-08_final_push.md` | 本轮收口与推送留痕 | 用于追溯本次公开仓库收口动作 |

## 五、维护原则

后续继续维护本仓库时，应默认遵守三个原则。首先，**不要把历史兼容字段当作公开默认文案**。其次，**不要把自动化巡检误写成自动归档**。最后，**不要再把阶段性检查单、临时 Todo 和最终主清单并行维护**。只要这三条原则被持续执行，后续即便再次出现上下文切换，也更容易恢复出准确的项目状态。
