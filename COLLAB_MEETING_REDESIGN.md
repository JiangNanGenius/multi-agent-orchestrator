# 协同讨论会议化重构方案

## 目标

当前协同讨论模块将被重构为一个**自动识别“正式会议 / 闲聊”场景**的专家会议系统。正式会议模式下，系统必须体现明确的会议流程，由主持人控场并指定发言专家，只有被主持人点名的专家才进入当前轮发言；同时系统需要持续生成会议纪要与结构化追溯记录。若识别为闲聊，则切换为更轻量、更自然的多人聊天模式，不强制套用正式会议流程。

## 新的双模式机制

| 模式 | 触发方式 | 行为特征 | 是否生成会议纪要 | 是否要求主持人控场 |
| --- | --- | --- | --- | --- |
| 正式会议 | 根据用户输入自动识别为会议议题，也允许前端显式偏向会议场景 | 有议程、有主持人、有阶段推进、有点名发言与收敛结论 | 是 | 是 |
| 闲聊模式 | 自动识别为轻松交流、咨询、陪聊或随意提问 | 发言更自然，不强制议程，不要求逐阶段推进 | 否，仅保留普通聊天记录 | 否 |

正式会议与闲聊的核心差异，不只是文案不同，而是**推进协议不同**。正式会议采用阶段化状态机；闲聊则采用轻量对话轮转。

## 正式会议流程

正式会议模式下，会话将引入固定流程阶段。建议第一版采用以下六段式结构。

| 阶段 | 标识 | 作用 | 典型产出 |
| --- | --- | --- | --- |
| 会议建立 | `meeting_init` | 确认议题、参会专家、主持人、会议目标 | 开场说明、会议目标 |
| 主持人开场 | `moderator_open` | 主持人说明议题背景、目标、讨论范围、规则 | 开场消息、议程说明 |
| 立场陈述 | `expert_statement` | 主持人按顺序点名 1~N 位专家发言 | 专家观点、关注点 |
| 交叉讨论 | `cross_discussion` | 主持人组织补充、质询、追问、回应 | 分歧点、补充意见 |
| 结论收敛 | `decision_sync` | 主持人总结共识、待办、风险与后续责任人 | 决议草案 |
| 会议结束 | `meeting_closed` | 生成会议纪要与可追溯记录 | 会议纪要、可追溯摘要 |

这意味着后端推进不再只是“生成一轮所有人发言”，而要根据当前阶段决定**该由主持人发言，还是由被点名专家发言，还是由主持人进入总结收敛**。

## 主持人机制

主持人不固定为总控中心。总控中心可以作为默认主持人，但系统必须支持用户在参会专家中指定任意角色担任主持人。主持人的职责是维持流程，而不是与普通专家并列自由发言。

| 能力 | 说明 |
| --- | --- |
| 主持人可切换 | 开会前可指定主持人，后续也可调整 |
| 主持人点名 | 当前轮只允许被主持人点到的专家发言 |
| 主持人收敛 | 在若干轮后输出当前共识、分歧与待决事项 |
| 主持人追问 | 对特定专家发起补充询问或澄清 |
| 主持人结束会议 | 触发纪要归档与阶段关闭 |

正式会议时，后端生成内容应优先产出一条主持人消息，说明当前点名了哪些专家，随后再生成对应专家消息。若进入结论阶段，则由主持人先做总结，再产出会议纪要。

## 自动识别机制

会话创建时，系统需要根据用户输入自动判断更像正式会议还是闲聊。第一版可以采用**规则优先 + LLM 辅助**的混合方式。

| 判断信号 | 倾向模式 |
| --- | --- |
| 出现“讨论方案、评审、开会、议题、风险、决策、安排、推进、复盘、计划”等词 | 正式会议 |
| 出现“聊聊、随便聊、闲聊、你们怎么看、轻松说说、吐槽”等词 | 闲聊 |
| 带明确目标、任务、风险、交付、角色协作要求 | 正式会议 |
| 语气松散、未设目标、主要为互动陪聊或泛问答 | 闲聊 |

如果规则无法明显判断，再调用一次轻量分类逻辑，返回 `meeting` 或 `chat`。

## 会议纪要与可追溯

正式会议必须可追溯，因此会话结构需要新增“会议记录”层，而不只是消息数组。建议新增以下数据域。

| 字段 | 说明 |
| --- | --- |
| `mode` | `meeting` 或 `chat` |
| `moderator_id` | 当前主持人 |
| `agenda` | 议题与会议目标 |
| `stage` | 当前会议阶段 |
| `stage_history` | 每次阶段推进记录 |
| `speaker_queue` | 当前轮被主持人点名的专家列表 |
| `minutes` | 实时更新的会议纪要 |
| `trace` | 结构化追溯日志，记录谁在何时因何发言 |
| `decision_items` | 当前形成的结论项 |
| `open_questions` | 尚未解决的问题 |
| `action_items` | 后续行动项 |

其中 `trace` 需要记录完整来源，例如“主持人在第 3 轮点名代码专家与部署专家就上线风险发言”，后续用户回看时才能知道每条发言是怎么进入会议流程的。

## 前端交互改造

前端页面需要从“讨论舞台”升级为“会议室”。第一版建议保留当前大致骨架，但改为更明确的会议控制区。

| 区域 | 改造方向 |
| --- | --- |
| 参会者选择区 | 增加“全选所有专家”按钮，并可指定主持人 |
| 顶部状态区 | 显示当前模式、主持人、当前阶段、当前议程 |
| 中控操作区 | 新增“点名发言”“进入下一阶段”“生成纪要”“结束会议” |
| 消息区 | 区分主持人消息、专家发言、纪要更新、结论项 |
| 侧边摘要区 | 实时显示会议纪要、待决事项、行动项 |

闲聊模式下，界面可以弱化这些会议控制元素，只保留多人聊天和参与者信息。

## 接口改造方向

现有 `start` 与 `advance` 接口需要扩展为支持会议模式。第一版建议在不大改路由名的前提下扩展请求体与返回体。

| 接口 | 新增关键字段 |
| --- | --- |
| `POST /api/collab-discuss/start` | `preferredMode`、`moderatorId`、`selectAll` |
| `POST /api/collab-discuss/advance` | `intent`、`speakerIds`、`stageAction`、`userMessage`、`constraint` |
| `POST /api/collab-discuss/conclude` | 可保留，但返回需包含 `minutes` 与 `decision_items` |
| `GET /api/collab-discuss/session/:id` | 返回完整 `mode`、`moderator_id`、`stage`、`minutes`、`trace` |

其中 `advance` 将不再只是“下一轮”，而是一个统一推进入口。它可以表达“主持人点名发言”“进入下阶段”“补充约束”“用户插话”“闲聊续聊”等不同推进意图。

## 后端生成逻辑改造方向

后端需要从“单轮批量多角色扮演”升级为“分阶段会议编排器”。第一版建议仍保留一次 LLM 调用生成若干消息的成本优势，但生成前必须先由后端决定本轮会议动作。

| 步骤 | 新机制 |
| --- | --- |
| 1 | 根据会话 `mode` 决定进入会议流还是闲聊流 |
| 2 | 若是会议流，读取 `stage`、`moderator_id` 与 `speaker_queue` |
| 3 | 若当前没有发言名单，则先由主持人选择或系统生成被点名专家 |
| 4 | 生成主持人消息，说明点名对象与讨论目标 |
| 5 | 仅生成被点名专家的消息，而非所有参与者都说话 |
| 6 | 更新 `minutes`、`decision_items`、`trace` |
| 7 | 判断是否需要进入下一会议阶段 |

这样既保留当前单次生成的实现便利性，又让系统在行为上真正像一个有主持人和流程的会议系统。

## 第一版落地原则

第一版不追求完美，而是优先让网页层面真正呈现“会议感”。因此建议先完成以下最小闭环：自动识别模式、可指定主持人、支持全选所有专家、正式会议按阶段推进、主持人点名专家发言、实时会议纪要、结束时可查看完整追溯记录。等这一版稳定后，再考虑加入投票、表决、附件引用、行动项派发等更复杂机制。
