# 本轮实施 Todo（按依赖优先级重排）

本清单不再按提出先后线性执行，而是按**底座能力 → 流转规则 → 前端展示 → 联调收尾**的依赖顺序推进。这样可以减少重复改动，避免前端先接字段、后端又返工，也便于把 `/new`、看门狗、冷热分层和飞书汇报统一挂到同一套任务账本上。

## 一、底座能力优先完成

这一组是所有后续能力的公共基础，必须先落地。

- [x] 吸收新的核心约束：以独立任务工作区和文件化账本作为跨 agent、跨任务的真实交接面。
- [x] 审查现有任务创建、持久化记录、前端展示与归档相关代码。
- [x] 设计任务工作区目录规范、任务索引规则与账本数据模型。
- [x] 实现任务创建时自动生成独立工作区与基础文件：`README.md`、`TODO.md`、`TASK_RECORD.json`、`HANDOFF.md`、`LINKS.md`、`STATUS.json`、`context/latest_context.json`、`ledger/*.jsonl`。
- [x] 实现任务账本更新机制，支持进度追加、交接记录、上下文刷新标记与任务链接。
- [x] 实现任务索引与归档索引，支持按任务代号、任务 ID、冷热状态和归档位置检索。
- [x] 增加 **超大项目冷热分层存储机制**：对经评估可能超过 `50GB` 的项目，创建时直接落到机械硬盘工作区，仅在固态盘保留必要元数据、索引与快速访问缓存。
- [x] 实现归档脚本，支持将已归档任务目录迁移到机械硬盘冷数据目录。
- [x] 增加 **归档后可回迁处理机制**：已归档到机械硬盘的项目应允许被 agent 重新激活，自动或按需回迁到固态盘继续处理，并同步更新任务状态、账本与前端展示。
- [x] 增加 **看门狗机制基础版**，优先实现为常驻或定时脚本，对任务流转进行巡检并记录健康状态。
- [x] 为看门狗定义 **修复动作**，支持补写 `TASK_RECORD.json` / `ledger`、触发重试、标记需要 `/new` 刷新、写入告警，并在必要时升级为通知。
- [x] 为看门狗补充 **可视化字段源数据**，至少输出最近巡检时间、最近异常、最近修复动作与当前健康状态。

## 二、在底座上补齐流转规则

这一组依赖前面的任务工作区与账本能力，完成后才能稳定跨任务、跨 agent 续接。

- [x] 研究并设计 **何时应该发送 `/new`** 的判断机制，形成可执行规则，并落地为系统逻辑与文件账本标记。
- [x] 把 `/new` 机制接入工作区上下文恢复链，确保刷新后可依靠 `README`、`HANDOFF`、`TODO`、`TASK_RECORD` 和 `latest_context` 恢复。
- [x] 增加 **小任务处理额外方案**，区分非标准大任务与轻量任务的创建、流转、留痕和归档策略。
- [x] 研究并实现 **agent 在飞书群内的汇报机制**，已落地为任务服务统一发送策略：按配置限定 agent 与事件，在任务创建、状态变更、进度追加、归档回迁与看门狗巡检节点汇报，并把发送结果回写到任务工作区 `feishu_reporting` 元数据与 `ledger/reports.jsonl`。
- [x] 研究看门狗应由 **独立 agent** 还是 **常驻/定时脚本** 实现，并形成最终落地方案；当前已确定以**独立脚本 / 定时任务**为第一落地点，脚本输出实现模式与调度建议，同时保留 `watchdog` agent 扩展位以兼容后续升级。
