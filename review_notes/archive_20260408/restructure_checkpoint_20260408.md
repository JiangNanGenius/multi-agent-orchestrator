## 五、已经识别出的关键风险

### 1. 当前仓库处于“部分文档已新命名、结构仍旧命名”的混合态

这意味着如果此时直接继续改某一个文件，而没有统一映射表和全仓替换策略，就很容易出现：

- 文档名和 agent 目录名不一致；
- 示例里是新称谓，但脚本参数仍要求旧 ID；
- 流程图、迁移脚本、测试样例还引用旧角色；
- 数据库/状态映射里仍然保留旧中文组织名。

### 2. 用户已经明确“不保留兼容”

这意味着后续不能再采取以下做法：

- 旧 ID → 新 ID 的别名映射；
- 文档中保留“历史名称也可用”；
- 脚本参数同时接受新旧两套；
- 数据表或迁移脚本中保留旧组织名仅供兼容。

后续正式重构时必须是**硬切换**。

### 3. 之前已改写的若干文档，仍然使用了“新称谓 + 旧 agent ID 目录”的混合表达

例如：

- 文案中已经出现“总控中心 / 规划中心 / 调度中心”等名称；
- 但代码与目录仍可能是 `taizi`、`zhongshu`、`shangshu`；
- 当前用户已经确认 `taizi` 不再保留，且新 ID 为 `centercontrol`。

所以后续要把这些“过渡性文案”再统一调整到**最终 ID 体系**之下。
