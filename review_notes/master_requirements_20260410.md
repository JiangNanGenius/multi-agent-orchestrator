# 本轮总需求整理（2026-04-10）

## 背景说明

本文件用于收口当前项目中分散在待办、核验记录、历史设计说明与界面复查文档里的需求项，避免后续继续以零散方式推进，导致重复修改、遗漏边界或验收口径不一致。

当前判断是：**项目已有一批页面入口和基础展示恢复，但仍存在若干“可见但未完全可用”的模块**。因此，本轮后续工作不再以“页面是否出现”为唯一标准，而以“功能是否闭环、交互是否完整、文案是否专业、状态是否可验证”为验收标准。

## 一、已形成明确共识的需求范围

| 需求域 | 本轮目标 | 当前状态判断 |
| --- | --- | --- |

| 账户安全 | 支持修改密码、修改用户名、重置默认凭据，并在设置中心可见 | 后端与前端链路已基本打通，仍需做最终验收 |

## 二、已确认基本完成或阶段性完成的事项

### 1. 设置中心增强

从设置联调记录与最近页面核验看，设置中心这条线已经完成了核心框架升级，不再是简单静态展示，而是已经进入可操作状态。当前已知已落位的能力包括：设置入口改为更集中的设置中心形式，系统参数读取与保存链路已打通，账户安全区已接入修改密码、修改用户名与默认凭据重置能力。

不过，这一项虽然不再属于“未开始”，但还不能视为完全收工。后续仍要做一次统一验收，重点核对以下几点：设置项是否全部真正影响页面行为、账户安全提示是否准确、异常反馈是否友好、以及设置中心内是否同步纳入语言偏好。

### 2. 顶部导航与基础页面可见性恢复

多份界面复查记录已经互相印证，目前顶部主菜单已恢复到较完整状态，包括任务看板、运行调度、AI开大会、Agent管理、模型配置、技能管理、快速任务、结果归档、模板中心、设置、全网搜索等入口。说明“页面消失”这一层的问题已基本解决，后续重点已转向“功能深度是否恢复”。

### 3. Agent 总览基础读取

较早前 Agent 总览曾因数据文件为空而异常，但验证记录显示该问题已通过重建总览与实时状态数据解决。此后页面已能展示节点总数、协作排行与详情卡片，因此基础数据读取层面已经恢复，不再是首要阻塞项。

## 三、当前最容易混乱、必须统一口径的未完成需求

### 1. 协作会议必须按“主持人版会议系统”收口

这一项已经查实：后端接入的不是旧式自由轮转讨论，而是你后续写的**主持人版会议实现**。相关设计文档也明确要求会议区分“正式会议 / 闲聊”双模式；正式会议具有主持人、阶段推进、点名发言、纪要、追溯、结论与行动项等结构化字段。

因此，前端后续一切会议改造都必须以此为准，不允许退回成普通聊天面板。会议模块的验收标准应至少包括以下几层：

| 会议能力 | 必须达到的标准 |
| --- | --- |
| 发起会议 | 可创建会议，支持选择参会者、主持人，并体现议题 |
| 会议状态 | 可看到模式、主持人、当前阶段、议程、最近动态 |
| 控制区 | 至少具备推进、暂停/恢复、结束、查看详情等核心入口 |
| 历史参照交互 | 对齐旧版 React `CourtDiscussion` 中的后台续会、接管本轮、返回会场等模式 |
| 结果沉淀 | 可查看纪要、阶段历史、追溯记录与关键结论 |

### 2. Agent 管理不能只停留在“只读演示”

当前 Agent 管理页已可打开，且基础数据能显示，但从现状看更接近“演示型总览”，而不是完整管理工作台。需要进一步梳理其与会议模块、监控模块、技能治理模块的边界，避免功能重复或入口混乱。

本轮建议将 Agent 管理的验收标准限定为：一是总览数据稳定可读；二是能清楚区分预置角色与普通专家；三是与会议/技能相关的联动入口不误导；四是文案专业统一。

### 3. 全局 Skill 仓库仍是关键未收口项

关于技能管理，当前文档里有两个表面上看似矛盾、实际可以统一的观察：一部分记录提到“全局 Skill 仓库容器已恢复，但读取失败”；后续复查则指出后端返回的全局 Skill 数量为 0，因此当前展示的更像空仓库，而不是接口报错。换言之，**入口已恢复，但数据语义仍需最终核实**。

这一项后续不能只看页面是否出现“全局 SKILL 仓库”标题，而要明确以下问题：到底是公开仓库本来为空、读取接口返回 0、还是前端映射仍有误；如果为空，是否需要演示数据；如果不应为空，则需要修接口或修请求逻辑。

### 4. 中英文界面切换需要作为正式需求恢复

用户已明确指出中英文切换功能缺失。当前已确认原先完整的语言状态、切换入口和文案映射不在现有静态看板里，因此这不是一个“小优化”，而是一个明确的功能缺口。

这项工作应拆成三层：第一，恢复头部或设置中心中的语言切换入口；第二，恢复语言偏好的本地或服务端持久化；第三，将主要导航、标题、关键按钮、空态提示、设置中心、会议面板等核心界面接入双语文案映射，而不是只切一个按钮标签。

### 5. 搜索页与任务发布入口还有独立遗留待办

从单独的搜索改造待办看，AI 搜索页与任务发布页还有一条未收口的需求线，包括移除 RSS / 订阅 / 外部推送遗留项、增加 Agent 驱动搜索任务入口、展示搜索专家忙闲状态、把任务目标从单选改为“自动分配 / 指定专家多选”互斥模式，并兼容 `targetDepts` 多选字段。

这一组需求虽然没有像“会议 / 设置 / 技能库”那样被反复点名，但如果目标是“这次一次尽量做完”，它不应被漏掉，应纳入统一总清单，至少明确为本轮是否一并处理。

## 五、本轮推荐的统一执行顺序

### 第一优先级：把已动过但未彻底收口的核心模块彻底闭环

优先完成设置中心最终复核、主持人会议控制台补齐、Agent 管理联动补强。这三项一旦闭环，用户此前反复指出的“缺了”“乱了”“只有展示没有交互”的核心问题会显著下降。

### 第二优先级：补齐跨页面的一致性能力

接下来恢复中英文切换，并统一头部、设置、会议、技能管理等模块中的主要文案结构。这样可以避免页面之间出现风格和语言状态不一致的问题。

### 第三优先级：解决技能库与搜索页这类容易在验收时被追问的遗留项

全局 Skill 仓库如果只是空数据，需要决定是否补演示数据；如果是读取链路问题，则要修复。AI 搜索页与任务入口则应至少完成一次“要不要纳入本轮最终收口”的明确决策与记录，避免再变成漏项。

## 六、建议采用的验收标准

| 模块 | 验收口径 |
| --- | --- |
| 设置中心 | 能读取、能保存、能持久化、能影响实际行为；账户安全操作有明确反馈 |
| 协作会议 | 不仅能看，还能创建、推进、恢复、结束，并能看到主持人流程与纪要追溯 |
| Agent 管理 | 数据稳定、角色边界清晰、入口不混乱、与会议/技能模块职责不冲突 |
| 技能管理 | 全局 Skill 区域不再停留在歧义状态，数据来源与空态含义明确 |
| 双语切换 | 入口存在、状态持久化、主要界面可切换而非局部伪双语 |
| UI / 文案 | 去除意义不明入口，专业化、统一化，不再出现明显临时性措辞 |
| 搜索 / 任务入口 | 如果纳入本轮，则至少完成入口收口与关键交互改造，否则需明确标注为下一轮 |

## 七、当前统一后的结论

本轮后续不应再以“哪里想到改哪里”的方式推进，而应以如下主线收口：**先完成设置中心最终复核与主持人会议控制台补齐，再处理 Agent 管理、全局 Skill、中英文切换与整体文案/UI，最后决定是否一并收口搜索页和任务入口遗留项。**

这份总表写好后，后续应该同步更新主待办文件，把每项标记为“已完成 / 进行中 / 待核验 / 待决定”，避免再次出现需求散落在不同记录里、结论彼此打架的情况。
