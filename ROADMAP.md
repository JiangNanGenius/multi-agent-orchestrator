# ROADMAP

> 这份路线图是公开的，但它不再把所有“值得做的方向”都当成本项目已经承诺要完成的事项。
>
> 从现在起，路线图分为三层：**已落地能力**、**已确认自有规划**、**外部启发候选方向**。其中只有前两层属于本项目正式范围；第三层仅表示可参考、可评估，不代表当前版本承诺。

---

## 一、已落地能力

> 这一部分只记录仓库中已经存在实现痕迹、并经过基本验证的能力。

- [x] 任务全生命周期管理（创建 → 预处理 → 规划 → 评审 → 派发 → 执行 → 汇总交付）
- [x] 任务叫停 / 取消 / 恢复
- [x] 结果归档系统（显式归档 / 批量归档 + 五阶段时间线）
- [x] 最终交付检查逻辑 —— 不通过则返修，无法自主决策时询问用户
- [x] 上下文窗口管理 —— 接近上限预警、超限压缩归档、续写衔接与交付前回读

## 二、已确认自有规划

> 这一部分只保留本项目已经明确决定推进、并愿意作为后续版本目标承担下来的事项。它们属于正式规划，但不代表当前版本已经全部完成。

### 1. 人工审批闭环

- **优先级**：中高

### 3. 知识沉淀与引用溯源
- [ ] 历史任务经验自动沉淀为可检索记录
- [ ] 相似任务检索与复用建议
- [ ] 交付报告引用溯源链
- **优先级**：中高

### 4. 部署校验固化
- [ ] 将部署前 AI 检查清单进一步落成脚本化或半自动执行入口
- [ ] 提供面向增量更新的标准校验流程封装（前端、看板、同步脚本、服务状态）
- [ ] 明确安装脚本、增量同步命令与运行时确认动作之间的职责边界
- **优先级**：中

## 四、如何理解“是否完成”

项目完成度后续只按以下口径判断：

| 口径 | 是否纳入完成度 |
|---|---|
| 已落地能力 | 纳入 |
| 已确认自有规划 | 作为后续版本差距参考 |
| 外部启发候选方向 | 不纳入 |

这意味着，后续在讨论“项目是否已经做好”“是否可以准备部署”时，不能再把所有参考性方向都算成本项目欠账。

## 五、治理入口与能力边界

为了避免公开路线图、README、阶段性留痕和真实实现再次出现口径分裂，后续治理应优先交叉查看以下长期入口：

| 文件 | 作用 |
|---|---|
| `README.md` | 公开主入口、部署路径与看板能力说明 |
| `docs/current_capability_boundary.md` | 当前真实能力边界、归档/交付/自动化巡检边界说明 |
| `todo.md` | 当前统一待办主清单 |
| `docs/project-progress-log.md` | 阶段性交付与改造留痕 |

其中，`docs/current_capability_boundary.md` 应作为后续恢复上下文时的首要校对入口；若 README、路线图或阶段性日志与其表述冲突，应先按该文档核对真实实现，再决定是否改写其它文档。

## 六、如何参与

1. 优先查看“**已确认自有规划**”，这些方向才是当前最适合继续投入的主线。
2. 如果你对“外部启发候选方向”有兴趣，建议先开 Issue 讨论是否正式采纳，再动手实现。
3. 提交 PR 前，请在描述里说明你修改的是哪一层能力：已落地修复、自有规划推进，还是候选方向探索。
4. 详细贡献方式见 [CONTRIBUTING.md](CONTRIBUTING.md)。

> 💡 没找到想做的方向？欢迎开 Issue 提议新功能，但提议会先经过“是否属于本项目正式路线”的判断，再决定放入哪一层。
