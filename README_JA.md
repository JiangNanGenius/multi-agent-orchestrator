## プロジェクト概要

**Multi-Agent Orchestrator** は、**役割分担**、**プロセス統治**、**結果の追跡可能性**を中心に設計されたマルチエージェント・タスクオーケストレーションプロジェクトです。中国語では**一个把复杂任务纳入可治理、可观测、可审查、可归档执行链路的多智能体协作编排框架**、英語では**a governed, observable, reviewable, and archivable orchestration framework for complex task execution** と要約できます。複数のエージェントを単に一つの会話に参加させるのではなく、タスク受理、計画、レビュー、ディスパッチ、専門実行という明確な責務分離を通じて、複雑な要求を管理可能かつ監査可能、さらに途中介入可能なパイプラインへ載せる構成になっています。[1] [2]

本プロジェクトには、リアルタイム・ダッシュボード、タスク状態の可視化、役割マッピング、テンプレート化されたタスク入口、成果アーカイブ、および関連自動化スクリプトも含まれています。そのため、マルチエージェント・アーキテクチャの参照実装としても、今後のプロダクト化に向けた実用的な土台としても利用できます。[1] [3]

## 説明の深化（2026-04-11）

初見の利用者が迷いにくいよう、以下の 3 点を運用上の優先原則として明記します。

1. **機能一覧より先にタスク主線を理解する**：受理 → 状態遷移 → 実行可視化 → アーカイブ継続、の流れを先に把握する。  
2. **全機能導入より先に可観測閉環を確認する**：タスク作成・状態更新・進捗記録・結果保管が確認できれば中核は稼働している。  
3. **拡張前に契約を統一する**：task ID 意味論、API 入口、運用 CLI を揃えてから画面や specialist を増やす。  

> この公開リポジトリは**脱敏済み公開版**です。ローカル実行データ、環境変数ファイル、内部レビュー記録、および一部のマシン側生成物は除外されており、アーキテクチャ、コード構成、UI 能力を公開展示するのに適した形へ整理されています。

> **出典メモ:** 本プロジェクトは現在、**JiangNanGenius** により継続整理・拡張されている **Multi-Agent Orchestrator** として案内されています。公開整理の過程で既存の開源実装や構成上の考え方を参考にした部分はありますが、現時点ではそれらを詳細な系譜説明として前面に出さず、必要最小限の謝辞とライセンス上の説明にとどめます。

> **現在のサポート境界:** 本プロジェクトでは **Docker イメージ、`docker-compose` 構成、コンテナ化デプロイ、および関連保守を提供しません**。公開版の正式な導入導線は **AI 支援デプロイを最優先** とし、保持しているインストールスクリプトは AI が環境診断後に必要と判断した場合のみ使う補助的な実行手段として扱います。

## プロジェクトフロー図

GitHub README に 1 枚の長いワークフロー図を埋め込むと、表示時に**細長くなりすぎて読みにくい**だけでなく、たとえ数枚に分けたとしても「ガバナンス主経路」「レビュー差し戻し」「ディスパッチ扇出」「結果収束」を同じ図に押し込めると、依然として密集して見えます。そのため、この公開版ではワークフローをさらに整理し、**5 枚の短くて見やすい分割図**として再構成しました。読者は「治理入口 → レビュー回路 → 実行収束 → デプロイ経路 → 看板運用閉環」の順で段階的に理解できます。

### 1. 治理入口の主経路

この図は、ユーザー入力からディスパッチ可能な状態に至るまでの主幹経路だけを残したものです。タスクは最初から specialist に直接落ちるのではなく、統一入口、総控、計画、レビューを経てから監督可能なディスパッチ段階へ進む、というガバナンス骨格を最初に把握できます。

![Multi-Agent Orchestrator 治理入口の主経路](docs/readme-governance-intake-flow.png)

> この図が答えるのは、**タスクがどのように統制システムへ入り、ディスパッチ可能な状態まで進むのか**という問いです。[1]

### 2. レビュー差し戻しと放行の仕組み

この図は、Planning Center と Review Center の往復関係だけを切り出して示します。承認、差し戻し、再修正、放行を主経路から分離することで、主フローに戻り矢印が重なって見づらくなる問題を避けています。

![Multi-Agent Orchestrator レビュー差し戻しと放行の仕組み](docs/readme-governance-review-loop.png)

> この図が答えるのは、**計画結果がどう審査され、不合格時にどう差し戻され、承認後にどうディスパッチへ進むか**という問いです。[1]

### 3. ディスパッチ、専門実行、結果アーカイブ

この図は、Dispatch Center が複数の専門ロールへ仕事を配分し、それぞれの成果を**コンテキスト整理・圧縮**、結果アーカイブ、最終納品チェック、そして最終納品へ収束させる部分だけを扱います。ここでは、以前は明示されていなかった二つの重要な制御を追加しています。第一に、**長時間タスクや多段階タスクは、その場の一時的な文脈だけに依存してはならず、納品経路へ入る前に再読可能な安定サマリーへ整理しておく必要があります。** 第二に、**結果はアーカイブ直後に完了扱いしてはならず、必ず最終納品チェックを通す必要があります。チェックで不合格なら修復ループへ戻り、自律的に判断できない論点やトレードオフがある場合は、ユーザー確認に戻ってから処理を継続します。** 前段の治理経路を同じ図から外したことで、扇出、コンテキスト収束、受け入れゲート、納品収束の構造がよりはっきり読めるようになりました。

![Multi-Agent Orchestrator ディスパッチ、専門実行、結果アーカイブ](docs/readme-governance-execution-flow.png)

> この図が答えるのは、**Dispatch Center が複数 specialist をどう組織し、コンテキスト整理、結果アーカイブ、最終チェック、修復分岐、ユーザー確認を経て、分岐した実行結果をどう単一の納品出口へ集約するか**という問いです。[1]

### 4. デプロイと検証のフロー

この図は、公開版で推奨するデプロイの考え方を説明します。重要なのは、最初からスクリプトを実行することではなく、**README を読み、環境を確認し、既存ランタイムの有無を判断し、AI に最小実行手順を設計させてから進める**ことです。`install.sh` と `install.ps1` は残していますが、必要性が明確な場合だけ使う補助入口です。

![Multi-Agent Orchestrator デプロイと検証のフロー](docs/readme-deployment-flow.png)

> 推奨順序は **README を読む → 環境を確認する → 増分接続か新規構築かを判断する → AI に最小コマンドを提案させる → 必要な場合のみスクリプトを使う → サービスと看板を検証する → 統制フローへ入る** です。

### 5. 看板運用とフィードバックのフロー

この図は、デプロイ後に日常運用でどのように看板を使って状態監視、タスク追跡、ディスパッチ調整、セッション確認、ログ確認、レポート化、アーカイブ化を行うかを示します。つまり、これは**運用時の可観測性とフィードバック閉環**を表す図です。

![Multi-Agent Orchestrator 看板運用とフィードバックのフロー](docs/readme-official-ui-flow.png)

## このプロジェクトが重要な理由

多くのマルチエージェント実装例は、複数モデルが同じ回答に参加すること自体に焦点を当てます。しかし本プロジェクトが重視するのは、**複数エージェントをどう統治するか**です。タスクは一つのエージェントへそのまま渡されるのではなく、まず統一入口で受理され、構造化された計画へ変換され、実行前にレビューされ、専用のディスパッチ層によって割り当てられ、最後に追跡可能な成果としてアーカイブされます。この設計により、本システムは長時間・多段階・途中確認が必要なワークフローにより適しています。[1]

以下の表は、本プロジェクトの設計上の重点を要約したものです。

| 観点 | 本プロジェクトでの扱い | 価値 |
| --- | --- | --- |
| タスク受理 | 統一入口と入力サニタイズ | ノイズを減らし、入力形式を標準化する |
| 計画 | 計画センターが目標と手順を分解 | 複雑タスクの実行可能性を高める |
| 品質管理 | 実行前にレビュー段階を挿入 | 不十分な計画がそのまま実行されるのを防ぐ |
| ディスパッチ | 専用のディスパッチセンターが割当と集約を担当 | 責任境界を明確にする |
| 可観測性 | ダッシュボードが状態・分布・心拍・ログを表示 | 監視、介入、デバッグを支援する |
| 結果保持 | アーカイブとレポート指向の表示を提供 | 振り返りと再利用を容易にする |

## システムアーキテクチャ

本プロジェクトは、**センター型の調整層と専門役割**によって協調を組織し、タスク状態機械とディスパッチロジックによって全体秩序を維持します。[1] [2]

```text
タスク受理
  ↓
総合制御センター
  ↓
計画センター
  ↓
レビューセンター
  ↓
ディスパッチセンター
  ↓
専門実行役割（データ / ライティング / エンジニアリング / コンプライアンス / デプロイ / Agent 管理 / 検索）
  ↓
結果アーカイブと納品
```

この構造の重要性は、**計画と実行の分離、レビューと実行の分離、ディスパッチと実行の分離**にあります。そのため、段階的推進、中間確認、明確な責任境界が必要なワークフローに自然に適応します。[1]

### 役割レイヤー

| レイヤー | 役割例 | 主な責務 |
| --- | --- | --- |
| 入口レイヤー | 総合制御センター | タスク受理、前処理、振り分け |
| 計画レイヤー | 計画センター | 要件理解、タスク分解、実行経路設計 |
| 検証レイヤー | レビューセンター | 計画品質の確認、差し戻し、リスク制御 |
| 編排レイヤー | ディスパッチセンター | 割当、協調組織、出力集約 |
| 実行レイヤー | データ、文案、エンジニアリング、コンプライアンス、デプロイ等 | 専門業務の実行 |

### 実行面

コード構造から見ると、本プロジェクトには二つの主要な実行面があります。一つは観測と操作を担当するダッシュボード側、もう一つはデータモデル、イベント通信、タスク編排を支えるオーケストレータ側です。[2] [3]

| 実行面 | 責務 | 技術形態 |
| --- | --- | --- |
| Dashboard 層 | タスク表示、状態監視、操作入口の提供 | HTML、フロントエンド成果物、Python サービス |
| Orchestrator 層 | タスクモデル、遷移、ディスパッチ、実行管理 | Python バックエンドとスクリプト |

## UI と体験上の特徴

マルチエージェントシステムを、単に動くだけでなく、見えて管理できるものにするため、本プロジェクトでは組織化された UI 設計に力を入れています。[3]

| パネル | 価値 |
| --- | --- |
| タスクカンバン | 状態別にタスク進行を観測できる |
| 実行監視 | システム負荷や役割活性度を素早く把握できる |
| 結果アーカイブ | 納品物とタスク出力の蓄積を確認できる |
| テンプレートセンター | 頻出タスクの標準起票を支援する |
| Agent 概要 | 各役割の状態とメタ情報を一元表示する |
| 協調ディスカッション | 複数視点で複雑な議題を深掘りできる |
| Feishu 返信診断表示 | 返信ポリシー、返信対象、スレッド文脈を可視化して回帖挙動を追跡できる |

このため、本リポジトリはデモ、PoC、社内イノベーション、教育用途において、理解しやすさと説明力を大きく高めます。

## クイックスタート

### 方法 1: AI 支援デプロイを優先する

推奨される最初の導線は、いきなり手動でリポジトリのスクリプトを実行することではありません。まず README とリポジトリ構成、現在の OpenClaw 実行環境をデプロイ段階の AI に読ませ、**初回導入なのか、増分更新なのか、どの最小実行経路を使うべきか**を先に判断させてください。

```bash
git clone https://github.com/JiangNanGenius/multi-agent-orchestrator.git
cd multi-agent-orchestrator
```

AI には最低限、次の項目を先に点検させることを推奨します。

| 点検項目 | 確認内容 |
| --- | --- |
| 実行環境 | OpenClaw が正常起動できるか、既存 workspace が壊れていないか |
| モデル設定 | 既存 Agent に利用可能なモデル認証が設定済みか |
| 導入タイプ | 初回接続か、フロントエンド / SOUL / 設定 / データの増分更新か |
| 実行入口 | 初回初期化スクリプトを使うべきか、段階的コマンドで進めるべきか |

AI の点検が完了した後は、その判断に従って初回初期化スクリプト、`bash start.sh`、または後述の増分手順のいずれかを選択してください。インストールスクリプトは常時の標準入口ではなく、AI が一括初期化に適すると判断した場合にのみ実行します。日常的な更新や部分反映では、前端ビルド、設定同期、データ更新のような増分経路を優先してください。

### 方法 2: ローカル開発

フロントエンドやバックエンドを継続開発したい場合は、モジュールごとに起動してください。

| モジュール | 推奨操作 |
| --- | --- |
| フロントエンド | `agentorchestrator/frontend/` で依存関係を導入しビルドする |
| 公式バックエンド | `agentorchestrator/backend/` を起動し、必要な API と静的配信設定を確認する |
| 認証確認 | 正式版フロントエンドからログインし、権限設定と初回導線が期待どおりか確認する |
| データ更新ループ | `bash scripts/run_loop.sh` を実行する |
| バックエンドコード | `agentorchestrator/backend/` でサービスや編排ロジックを拡張する |

### 方法 3: 公開展示リポジトリとして利用する

このリポジトリは公開版として整理済みです。GitHub で公開を継続する場合は、ローカル実行データ、`.env` ファイル、実際の Webhook、Token、マシン側ログをコミットしないでください。

multi-agent-orchestrator/

## オープンソース公開方針

本リポジトリを公開する際には、次の原則を維持することが推奨されます。第一に、アーキテクチャ、コード、文書、デモ可能な UI は公開すること。第二に、ローカル実行ファイルと機微設定は除外すること。第三に、実際の外部チャネル資格情報はプレースホルダへ置き換えること。第四に、README 内でデモ用要素と利用者側設定が必要な要素を明確に区別することです。

| 種別 | 現在の推奨 |
| --- | --- |
| 環境変数 | サンプルのみを残し、実値はコミットしない |
| 実行データ | ローカル `data/` ディレクトリをコミットしない |
| 内部レビュー資料 | 内部メモや一時資料をコミットしない |
| 外部チャネル | プレースホルダまたは説明のみ残す |
| リポジトリ表現 | 公開・専門的・理解しやすい状態を保つ |

### 公開前に推奨される最小検証

公開リポジトリへ push する前には、変更内容に見合った最低限の検証を一度は通し、文章だけの自己確認で終わらせないことを推奨します。

| 変更種別 | 推奨される最小検証 |
| --- | --- |
| 文書のみの更新 | 見出し構造、リンク、画像パス、版歴、および中国語・英語・日本語 README 間の整合を確認する |
| ダッシュボード / フロントエンド変更 | `agentorchestrator/frontend/` でビルドを実行し、主要画面が正常に読み込まれることを確認する |
| Python サービス / スクリプト変更 | 構文確認と、影響範囲に応じた安全な実行経路の再確認を行う |
| 脱敏や公開整理 | secrets、webhook、ログ、runtime snapshot、内部メモ、ローカルマシンのパス残留を再走査する |

### 公開 push 前の脱敏チェックリスト

公開 push の直前には、以下のような内容が誤ってコミット対象へ含まれていないかを再確認してください。編排系リポジトリでは、これは見栄えの問題ではなく、公開境界そのものの一部です。

| 公開リポジトリへ含めてはいけないもの | リスク |
| --- | --- |
| 実値入り `.env`、API key、webhook、DB パスワード | 即時の資格情報漏洩 |
| ローカル runtime log、task snapshot、実行アーカイブ | 内部タスク内容や運用意味論の漏洩につながる |
| マシン固有パス、cache、build の一時廃棄物 | 環境情報の露出とリポジトリ衛生の低下 |
| 内部レビュー用メモ、一時 TODO スクリーンショット、未脱敏デモデータ | 公開成果物の境界を超える |

外部コントリビューターへ協作入口を説明する場合は、[`CONTRIBUTING.md`](./CONTRIBUTING.md) と [`SECURITY.md`](./SECURITY.md) を併せて参照し、協作経路、最小検証の期待値、脱敏境界を同じ基準で保つことを推奨します。

## よくある問題と確認入口

既存環境へこのリポジトリを接続する場合や、公開版を基礎としてさらに製品化を進める場合には、次のような問題から優先的に切り分けることを推奨します。インストールスクリプトを何度もやり直すよりも、問題が**文書方針・フロントエンド build・実行設定・データ同期**のどこに属するかを先に見極める方が効率的です。

| 症状 | 先に確認すべき点 | 推奨アクション |
| --- | --- | --- |
| フロントエンド build が失敗する | `agentorchestrator/frontend/` の依存関係、Node バージョン、最近の二言語化変更で型エラーが入っていないか | まず build 回帰を実行し、直近の変更ファイルを追う |
| ダッシュボードは開くが内容が空 | backend API、同期スクリプト、統計更新手順が実行済みか | 先に設定同期とデータ更新をやり直し、その後 API 状態を確認する |
| 初回ログインやパスワード変更が不安定 | 既定アカウント導線、認証状態、ブラウザ cache | 強制される初回パスワード変更導線を確認してから、フォームと backend を点検する |
| Agent は登録されたがモデルを呼べない | 主 Agent に有効なモデル認証があるか、認証同期を実行済みか | まず主認証を整え、その後に認証同期または初期化再実行を選ぶ |
| 英語切替後も中国語や旧称が見える | 対象パネルが locale 層に接続済みか、互換値が UI に漏れていないか | `agentorchestrator/frontend/src/i18n.ts` と該当パネルを先に確認する |
| 公開 push してよいか判断できない | ログ、snapshot、secret、マシン固有パス、内部メモが残っていないか | 公開前チェックリストに沿って脱敏レビューを再実施する |

## 公開保守の推奨リズム

この公開リポジトリを継続的に保守する場合は、文書、UI、スクリプト、公開メタ情報を一つの公開チェーンとして扱うことを推奨します。どれか一部だけを更新すると、README、GitHub のリポジトリ概要、ダッシュボード表示、実際の実行経路が再び乖離しやすくなります。

| 保守アクション | 推奨頻度 | 目的 |
| --- | --- | --- |
| 3 言語 README を同期する | 公開可視な変更のたび | 公開入口、フロー図、デプロイ方針を一致させる |
| `todo.md` の留痕を更新する | 大きな改造ごと | 次回の継続作業に必要な文脈を残す |
| `CONTRIBUTING.md` / `SECURITY.md` を再確認する | 協作方針や公開境界が変わるたび | 貢献入口と脱敏基準を揃える |
| フロー図とディレクトリ説明を見直す | タスク流れやモジュール境界が変わるたび | README 内のアーキテクチャ説明の陳腐化を防ぐ |
| 公開 push 前に脱敏再確認を行う | push のたび | 機微情報、runtime 文脈、一時ファイルの誤コミットを防ぐ |

> **推奨する保守原則：** まず「公開して読める」「デプロイできる」「後から検証できる」を守り、その上で見た目の完成度を磨いてください。編排型プロジェクトでは、安定した文書口径そのものが品質の一部です。
