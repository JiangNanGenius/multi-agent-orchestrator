## 项目简介

**Multi-Agent Orchestrator** 是一个围绕“多角色协作、流程治理、结果可追溯”设计的多智能体任务编排项目。它并不把多个 Agent 简单拼接成一个聊天群，而是通过任务入口、规划中心、评审中心、调度中心和专业执行角色之间的明确分工，把复杂任务纳入一条可管理、可审计、可干预的执行链路。[1] [2]

项目同时提供实时 Dashboard、任务流转状态、角色分工映射、模板化任务入口、结果归档和部分自动化脚本，使其既能作为一个多 Agent 架构样例仓库，也能作为后续产品化改造的基础工程。[1] [3]

> 本公开仓库为**脱敏整理版本**，已排除本地运行态数据、环境变量文件、审查记录和部分机器侧生成内容，适合公开展示架构设计、核心代码组织方式与界面能力。

> **主要引用说明：** 本项目的公开整理与当前展示版本，主要引用并参考了 [`wanikua/danghuangshang`][4] 的公开实现思路、组织方式与许可要求；同时结合 [`cft0808/edict`][5] 的现有代码基础进行了现代化整理、命名收口与公开发布改写。相关引用已在本文档与仓库说明中明确保留。

## 为什么这个项目值得关注

大多数多智能体示例更关注“让多个模型参与回答”，而这个项目更强调**流程控制**。在这里，任务不是直接交给单个 Agent 独立完成，而是先被统一受理，再进入规划、核验、派发与执行，最终形成结果归档。这样的设计使系统更适合处理复杂、长链路、需要中途检查的任务场景。[1]

下表概括了本项目的设计重点。

| 维度 | 本项目的处理方式 | 价值 |
| --- | --- | --- |
| 任务受理 | 统一入口接收与清洗任务 | 降低输入噪声，保证任务格式一致 |
| 方案规划 | 由规划角色拆解目标与步骤 | 提高复杂任务的可执行性 |
| 质量核验 | 在执行前引入评审节点 | 减少错误方案直接进入执行层 |
| 调度编排 | 由专门角色负责派发与汇总 | 让执行责任清晰、边界明确 |
| 实时观测 | Dashboard 展示状态、分布、心跳与日志 | 便于监控、介入与排障 |
| 结果沉淀 | 提供归档与报告化视图 | 方便复盘与复用 |

## 核心能力

项目的核心价值不只在于“多 Agent”，更在于**多 Agent 如何被治理**。当前公开版本已经呈现出较完整的系统轮廓，包括结构化任务流转、看板化观测界面、角色配置与模板化入口等关键能力。[1] [2] [3]

| 模块 | 说明 | 对应位置 |
| --- | --- | --- |
| 任务总览 | 以看板方式展示任务状态、过滤与检索 | `dashboard/` |
| 运行监控 | 展示任务分布、角色活跃度与运行状态 | `dashboard/dashboard.html` |
| 结果归档 | 查看完成任务与阶段性输出 | `dashboard/` |
| 协同讨论 | 支持围绕议题进行多角色讨论 | `dashboard/court_discuss.py` |
| 模板中心 | 通过预设模板快速发起任务 | 前端界面与 `dashboard/` |
| 模型/技能管理 | 为角色配置能力边界与运行策略 | `scripts/`、前端界面 |
| 后端编排 | 提供任务模型、调度逻辑与事件链路 | `edict/backend/` |
| 安装与启动 | 提供一键安装与服务启动脚本 | `install.sh`、`start.sh` |

## 系统架构

项目以“中心角色 + 专业角色”的方式组织多 Agent 协作，并通过任务状态机和调度逻辑维持整体秩序。[1] [2]

```text
任务入口
  ↓
总控中心
  ↓
规划中心
  ↓
评审中心
  ↓
调度中心
  ↓
专业执行角色（数据 / 文案 / 代码 / 合规 / 部署 / Agent 管理 / 情报）
  ↓
结果归档与交付
```

这一结构的关键意义在于：**规划与执行分离、评审与执行分离、调度与执行分离**。因此，系统天然适合承载需要多步骤推进、需要中间检查、需要明确责任边界的任务流程。[1]

### 角色分层

| 层级 | 角色示例 | 主要职责 |
| --- | --- | --- |
| 入口层 | 总控中心 | 接收任务、预处理、分流 |
| 规划层 | 规划中心 | 理解需求、拆解任务、设计执行路径 |
| 核验层 | 评审中心 | 校验规划质量、退回修订、控制风险 |
| 编排层 | 调度中心 | 派发任务、组织协作、汇总输出 |
| 执行层 | 数据、文案、代码、合规、部署等角色 | 完成具体专业工作 |

### 运行形态

从代码结构看，本项目包含两类主要运行面：一类是用于观测和操作的 Dashboard 端，另一类是承载数据模型、事件通信和任务调度的后端部分。[2] [3]

| 运行面 | 职责 | 技术形态 |
| --- | --- | --- |
| Dashboard 层 | 展示任务、监控状态、提供交互入口 | HTML + 前端构建产物 + Python 服务 |
| Orchestrator 层 | 管理任务模型、流转、调度与执行 | Python 后端与脚本 |

## 适用场景

这个项目适合以下几类使用场景。第一类是作为**多 Agent 编排架构参考实现**，帮助团队理解多角色分工、任务状态流转和中间审查机制如何落地。第二类是作为**智能流程平台原型**，进一步接入数据库、消息系统、外部工具或企业 IM。第三类则是作为**教学与研究样例**，用于演示多智能体协作系统如何从 Prompt 设计走向流程设计。[1] [2]

| 场景 | 使用方式 |
| --- | --- |
| 架构研究 | 研究多角色协作与可观测编排 |
| 产品原型 | 作为智能工作流平台雏形继续扩展 |
| 企业内部试点 | 对接内部任务流、审批流或知识库 |
| 教学演示 | 展示多 Agent 系统设计方法 |
| 二次开发 | 基于现有前后端结构继续演进 |

## 快速开始

### 方式一：直接体验界面

如果你主要想快速理解项目形态，可以先运行 Dashboard 与相关脚本，观察任务面板、协同讨论、归档等能力。[3]

```bash
git clone <your-repo-url>
cd multi-agent-orchestrator
bash start.sh
```

随后在浏览器打开：

```text
http://127.0.0.1:7891
```

### 方式二：本地开发

如果你希望继续开发前端或后端，可以按模块分别启动。

| 模块 | 建议操作 |
| --- | --- |
| 前端 | 在 `edict/frontend/` 中安装依赖并构建 |
| Dashboard 服务 | 运行 `python3 dashboard/server.py` |
| 数据刷新 | 运行 `bash scripts/run_loop.sh` |
| 后端代码 | 在 `edict/backend/` 中继续扩展服务层与调度逻辑 |

### 方式三：作为公开展示仓库使用

当前仓库已经整理为适合公开发布的版本。若你要继续发布到 GitHub，建议优先保持以下边界：不要提交本地运行态数据，不要提交 `.env` 文件，不要提交真实 webhook、token 或机器侧日志。

## 项目结构

仓库结构体现了项目的核心分层：前端展示、Dashboard 服务、后端编排、Agent 人格模板与自动化脚本相互配合。[2] [3]

```text
multi-agent-orchestrator/
├── agents/                 # 多角色 Agent 的 SOUL / 配置模板
├── dashboard/              # Dashboard 页面、服务端与协同讨论模块
├── docs/                   # 架构、上手、补充说明文档
├── edict/
│   ├── backend/            # 任务模型、服务层、worker 等后端代码
│   └── frontend/           # React 前端工程
├── scripts/                # 安装、同步、技能、刷新与辅助脚本
├── tests/                  # 测试代码
├── install.sh              # 安装脚本
├── start.sh                # 启动脚本
└── edict.service           # 部署服务配置
```

### 关键目录说明

| 路径 | 说明 |
| --- | --- |
| `agents/` | 多角色能力定义与人格模板 |
| `dashboard/` | 运行看板、服务接口与讨论能力 |
| `docs/` | 架构与使用说明文档 |
| `edict/backend/` | 编排逻辑、任务模型、worker 机制 |
| `edict/frontend/` | React 前端源代码与构建产物 |
| `scripts/` | 安装、同步、刷新、管理脚本 |
| `tests/` | 自动化测试与一致性校验 |

## 开源发布说明

本仓库面向公开展示时，建议保持以下原则：一是公开结构、代码、文档和可演示页面；二是移除所有本地运行态文件与敏感配置；三是用示例值代替真实外部通道凭据；四是通过清晰 README 说明哪些内容属于演示用途，哪些内容需要自行配置后才能运行。
